# System 模块说明

## 定位

`system` 模块是整个项目的基础内核，负责未来所有业务模块共用的基础能力。

当前已经承载：

- 登录认证
- 用户与角色
- 菜单与动态路由
- 功能权限与 API 注册
- 作用域
- 团队与租户上下文
- 团队成员和团队功能边界

后续新增业务模块时，应尽量复用这套基础能力，而不是重新实现一套权限和上下文体系。

## 当前模块划分

- `auth/`
  登录、用户信息、当前权限集合
- `user/`
  用户管理、用户角色、用户级功能权限覆盖
- `role/`
  角色、菜单权限、角色功能权限、角色数据权限
- `menu/`
  菜单树、菜单管理、运行时菜单返回
- `permission/`
  功能权限注册表
- `apiendpoint/`
  API 注册表与接口元数据同步
- `scope/`
  作用域定义
- `tenant/`
  团队、当前团队、团队边界、团队成员

## 基础判断顺序

当前系统建议统一按以下顺序理解权限：

1. 用户状态
2. 团队成员状态
3. 当前上下文
4. 团队功能边界
5. 用户个人覆盖
6. 角色权限
7. 默认拒绝

## 未来扩展原则

- 业务模块尽量复用 `system` 的认证、权限、租户上下文和 API 注册能力
- 新业务不需要重新设计一套权限模型，只需要补：
  - 菜单入口
  - 功能权限
  - 接口元数据
  - 租户数据隔离
- 数据权限当前已具备配置基础，但默认不作为新增业务接入的第一优先级
