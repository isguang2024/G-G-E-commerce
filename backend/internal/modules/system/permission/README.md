# 功能权限说明

## 1. 权限分层

系统里有三层权限，不要混用：

- 菜单权限：
  决定菜单与页面入口是否显示
- 功能权限：
  决定按钮和接口是否允许执行
- 数据权限：
  决定数据范围

当前开发重点是菜单权限和功能权限。数据权限暂时保留配置基础，但默认不参与业务查询过滤。

## 2. 功能权限唯一标识

功能权限的真实唯一标识是：

- `resource_code`
- `action_code`
- `scope_id`

也就是同一业务动作在不同作用域下可以同时存在，但必须作为不同权限定义处理。

示例：

- `report:list@global`
- `report:list@team`

## 3. 作用域使用规则

当前只保留两类作用域：

- `global`
- `team`

作用域的含义是“这条权限定义属于哪一层”，不是菜单分组，也不是业务分类。

如果一个功能同时支持平台和团队使用：

1. 权限定义拆成两条
2. 路由入口可以拆成两套薄入口
3. 业务实现尽量复用同一套 service

不要让一个接口同时混跑多个作用域语义。

## 4. 权限来源

当前来源统一为三类：

- `api`
  接口元数据自动注册
- `system`
  系统内置种子
- `business`
  业务定义权限

同时还用 `feature_kind` 区分：

- `system`
- `business`

这两个字段不要混为一谈：

- `source` 说明权限进入系统的方式
- `feature_kind` 说明权限属于系统功能还是业务功能

## 5. 模块归属

功能权限配置页的主分组，应按以下顺序理解：

1. `feature_kind`
2. `module_code`
3. `resource_code:action_code`

`module_code` 是业务模块归属，不要再让 `category` 或 `scope` 承担这个职责。

## 6. 角色、用户、团队、成员四层关系

### 角色功能权限

角色是基础授权层。

### 用户功能权限

用户页不是重新配一整套权限，而是“平台级个人例外覆盖层”。

用户单条权限只应有三态：

- `继承角色`
- `单独允许`
- `单独拒绝`

### 团队功能权限

团队页配置的是“团队功能边界”。

团队没开通的团队作用域功能，团队成员和团队管理员都不能用。

### 团队成员功能权限

团队成员页配置的是“团队内个人例外覆盖层”。

## 7. 冲突规则

当前推荐规则：

- 用户状态优先于一切
- 团队成员状态优先于角色和覆盖
- 团队边界高于团队内角色与个人授权
- 单用户覆盖高于角色权限
- `deny > allow > inherit`

如果角色未配置，而用户选择“继承角色”，最终结果是无权限。

## 8. 菜单与功能权限的边界

当前项目不建议引入完整的“用户级菜单权限表”。

推荐做法：

1. 菜单继续由角色菜单权限控制
2. 菜单可附加基础功能门槛
3. 用户若被单独关闭该基础功能，则菜单可同步隐藏

当前菜单 `meta` 推荐约定：

- `requiresTenantContext`
- `requiredAction`
- `requiredActions`
- `actionMatchMode`
- `actionVisibilityMode`

这样可以让“角色菜单 + 用户功能覆盖”自然联动，而不必再维护一套用户级菜单权限。

其中：

- `actionMatchMode`
  - `any` = 任意满足
  - `all` = 全部满足
- `actionVisibilityMode`
  - `hide` = 功能门槛不满足时隐藏菜单，并阻止路由进入
  - `show` = 功能门槛不满足时仍显示菜单，不额外拦菜单入口，只由页面按钮和后端接口继续控制

## 9. API 注册约定

每个需要进入权限体系的接口，都应在路由注册时补全元数据：

- `resource_code`
- `action_code`
- `scope`
- `feature_kind`
- `module`

接口注册用于两件事：

1. 自动同步 API 注册表
2. 自动补齐或更新功能权限注册表

## 10. 新功能接入最小清单

以后新增一个功能时，按下面顺序处理：

1. 定义 `feature_kind`
2. 定义 `module_code`
3. 定义 `resource_code`
4. 定义 `action_code`
5. 定义 `scope`
6. 接入 API 注册
7. 前端按钮使用 `v-action`
8. 后端接口使用 `RequireAction`
