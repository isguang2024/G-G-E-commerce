# 团队与租户说明

## 1. 定位

当前项目里的“团队”就是租户上下文。

系统里同时存在两类团队相关能力：

- 平台侧团队管理
- 当前团队上下文管理

二者都需要保留，但不能混淆。

## 2. 两类接口的边界

### 平台侧接口

面向系统管理员或平台角色，用来管理任意团队。

典型路径：

- `/api/v1/tenants`
- `/api/v1/tenants/:id`
- `/api/v1/tenants/:id/members`
- `/api/v1/tenants/:id/actions`

特点：

- 不依赖当前团队上下文
- 作用域一般为 `global`

### 当前团队接口

面向当前团队管理员或团队成员。

典型路径：

- `/api/v1/tenants/my-team/...`

特点：

- 依赖当前团队上下文
- 作用域一般为 `team`

## 3. 团队相关数据职责

### `tenant_members`

负责：

- 用户是否属于某团队
- 成员状态
- 加入时间
- 快速身份标识

不要把复杂角色权限继续堆在这里。

### `user_roles`

负责：

- 用户在平台上下文下有哪些角色
- 用户在当前团队下有哪些角色

语义：

- `tenant_id IS NULL`：平台角色
- `tenant_id = 当前团队`：团队角色

### `tenant_action_permissions`

负责：

- 某团队开通了哪些团队作用域功能

这里是团队边界，不是团队成员的个人授权。

## 4. 团队边界的判断规则

团队边界只对团队作用域功能生效。

判断方式：

- 当前 `tenant_id`
- 当前 `action_id`
- 是否存在于 `tenant_action_permissions`

存在：

- 已开通

不存在：

- 未开通

当前约定已经收紧：

- 如果某团队没有任何边界记录，不再视为全开
- 而是视为“全部未开通”

## 5. 当前团队功能的判定顺序

团队作用域下，推荐统一按以下顺序判断：

1. 用户全局状态
2. 当前团队成员状态
3. 团队边界是否开通
4. 当前用户团队内个人覆盖
5. 当前用户团队角色权限
6. 默认拒绝

## 6. 系统管理员与团队管理员

系统管理员和团队管理员可以是同一个人，但接口语义不能混用。

规则：

- 走平台侧接口时，按平台权限判断
- 走 `my-team` 接口时，按当前团队上下文判断

即使同一用户同时拥有两类身份，也不要让一个接口同时混跑两套语义。

## 7. 菜单与团队上下文

团队上下文页面应继续使用：

- `requiresTenantContext = true`

普通用户无团队时：

- 团队菜单不显示
- 直接访问 URL 也要被守卫拦截

系统管理员页面不要误用这个标记。

## 8. 未来新增业务的租户约定

以后新增商品、订单、内容等团队业务时：

1. 数据表默认加 `tenant_id`
2. 页面入口区分平台入口和团队入口
3. 团队内接口使用团队作用域功能权限
4. 平台审核或平台管理接口使用全局作用域功能权限

不要把“前台/后台”作为边界，而要按“平台入口 / 团队入口 / 数据范围”建模。
