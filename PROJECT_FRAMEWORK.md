# 项目主框架定义与执行清单

## 1. 文档定位

本文件用于定义项目总体框架、核心边界和执行清单。

这里描述的是长期稳定、跨模块通用的约束，不展开具体实现细节，不替代模块内 README、接口文档或迁移说明。

## 2. 项目主框架定义

### 2.1 项目定位

本项目不是单一业务页面集合，而是一套可持续扩展的电商业务基础框架。

当前应优先沉淀并复用以下共性能力：

- 认证与身份识别
- 用户、角色与权限体系
- 菜单、功能点与 API 注册
- 团队与租户上下文
- 可复用的前后端模块组织方式

新增业务默认接入既有基础层，不重复建设第二套权限、租户或上下文体系。

### 2.2 模块边界

系统基础能力与业务能力分层演进。

- `system` 模块承载公共基础能力
- 业务模块在统一边界内扩展
- 公共能力优先下沉，业务差异在模块内收敛

涉及平台侧与团队侧时，语义必须分开，入口可以分开，实现可以复用，但作用域不能混淆。

### 2.3 架构分层

项目按前后端分层协作，并在各层内部保持稳定职责边界。

- 后端负责领域模型、接口能力、权限判定、租户语义和数据治理
- 前端负责页面编排、交互承载、状态组织和权限结果消费
- 文档负责沉淀长期稳定的模块说明、边界约束和接入方式

总体上应避免把临时实现细节上升为框架约束，也避免把框架级规则散落在单点代码里。

### 2.4 前后端协作原则

- 后端负责权限、菜单、接口元数据和作用域语义的真实来源
- 前端负责基于后端结果进行展示、交互和权限收敛
- 类型、接口、菜单、按钮权限应保持同一语义链路

前端不应通过静态补齐方式绕过后端定义，后端也不应把前端语义依赖写成隐式约定。

### 2.5 模块组织原则

模块按统一结构持续演进，目录和职责应保持可预期。

- 后端模块保持稳定的模块入口、接口层、业务层和数据访问层边界
- 前端按 API、类型、页面、组件、状态等维度组织
- 公共能力优先沉淀到公共模块或基础层，避免在业务模块中横向复制

模块组织的目标是降低新增业务和后续维护成本，而不是追求一次性设计完整。

### 2.6 数据与权限原则

- 菜单权限用于控制导航和页面入口
- 功能权限用于控制按钮和接口动作
- 团队数据默认按租户隔离
- 数据迁移与结构变更分开治理

所有进入权限体系的接口，都应具备清晰、可追踪、可补齐的元数据定义。

### 2.7 API 与迁移治理原则

- 接口不仅提供业务能力，也承担注册、权限识别和边界声明职责
- 表结构演进与数据修复应分开治理
- 迁移策略应服务于长期维护，而不是短期补丁堆积

所有结构变更、历史修复和索引整理都应具备清晰归属，避免职责混用。

### 2.8 文档分层原则

- 根目录文档只保留总体约束、稳定入口和全局规则
- 模块细节写在对应模块文档中
- 阶段性设计、过渡方案和一次性讨论不应长期堆积在总纲文档

总纲负责定义方向和边界，模块文档负责说明实现和接入。

## 3. 执行清单

以下清单用于新增模块、改动核心流程或联动前后端时的最低自检。

### 3.1 新增或调整业务时

- 是否复用了已有认证、权限、租户与上下文能力
- 是否遵循既有模块边界，而不是引入平行体系
- 是否区分平台作用域与团队作用域

### 3.2 调整模块结构或公共能力时

- 是否影响既有模块边界、目录约定或公共能力归属
- 是否应沉淀为基础能力，而不是停留在单个业务模块中
- 是否同步更新对应模块文档入口

### 3.3 调整接口或权限时

- 是否同步检查菜单权限、功能权限与 API 元数据
- 是否保证前后端权限标识语义一致
- 是否避免一个接口混跑多个作用域

### 3.4 调整数据结构或迁移时

- 是否区分表结构变更与数据修复类迁移
- 是否保证迁移具备幂等性
- 是否评估团队数据隔离和历史数据兼容性

### 3.5 新业务接入时

- 是否补齐模块入口、基础数据结构和必要文档
- 是否接入菜单、功能权限、接口元数据和租户隔离
- 是否确认前后端链路具备一致的业务语义

### 3.6 提交前最小确认

- 是否补齐了对应模块文档或入口说明
- 是否确认影响范围覆盖前端、后端和权限链路
- 是否说明已验证项、未验证项和后续待处理事项

## 4. 文档使用规则

- 总体原则写在本文件
- 模块细节写在各模块自身文档
- 临时方案、阶段性设计和一次性讨论不沉淀到本文件

如果具体实现与本文件冲突，应优先回到模块设计文档和真实代码结构中核对，再决定是否更新本文件。
