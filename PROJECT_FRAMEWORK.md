# 项目主框架定义与执行清单

## 1. 文档定位

`PROJECT_FRAMEWORK.md` 只定义全局长期约束，不承载具体模块实现细节。

适用场景：
- 新增业务模块
- 调整权限链路
- 变更迁移策略
- 涉及前后端联动的核心流程改造

不适用场景：
- 单页样式微调
- 局部函数重命名
- 一次性临时排障说明

## 2. 主框架

### 2.1 项目目标

本项目是“可持续扩展的电商基础平台”，不是页面集合。

必须复用且持续演进的底座能力：
- 认证与身份
- 角色/权限/功能包
- 菜单与 API 元数据
- 平台与团队上下文
- 前后端统一语义链路

### 2.2 分层边界

- 后端：领域模型、授权判定、快照与迁移治理。
- 前端：页面编排、状态组织、权限结果消费、交互一致性。
- 文档：稳定规则、接入方式、变更约束。

禁止：
- 前端绕过后端语义自行补权限。
- 后端依赖前端约定做隐式逻辑。
- 在业务模块复制一套平行权限体系。

### 2.3 模块边界

- `system` 承载公共基础能力。
- 业务模块在 `system` 规则内扩展。
- 平台(`platform`)与团队(`team`)语义必须显式隔离，允许复用实现，不允许混用作用域。

### 2.4 权限与数据原则

- 菜单权限控制入口。
- 功能权限控制动作/API。
- 团队数据默认租户隔离。
- 结构迁移与数据修复分离治理。

### 2.5 API 与迁移治理

- API 不仅是业务能力，也是权限识别与边界声明载体。
- 迁移必须幂等。
- 旧兼容逻辑默认下沉到迁移阶段，运行时只保留最小必要兼容层。

## 3. 执行清单（可执行）

### 3.1 触发条件

出现以下任一项时，必须执行本清单：
- 修改核心模型、权限链路、菜单链路、API 注册、迁移逻辑。
- 联动前后端两端改造。
- 新模块接入系统能力。

### 3.2 输入检查

提交前必须明确：
- 变更所属上下文：`platform` / `team` / `platform,team`。
- 变更是否影响：菜单、功能权限、API 元数据、快照刷新。
- 是否包含：表结构变更、数据修复、兼容逻辑下线。

### 3.3 设计检查

- 是否复用既有认证/权限/租户能力。
- 是否没有引入平行体系。
- 是否明确影响范围（后端、前端、文档、迁移）。
- 是否把“临时补丁”收敛为“可维护机制”。

### 3.4 实施检查

- 接口：语义一致，权限标识前后端一致。
- 数据：迁移幂等，可回放，历史环境可升级。
- 权限：菜单、功能权限、API 元数据链路完整。
- 文档：新增或修改的稳定规则已同步。

### 3.5 完成标准

满足以下条件才算完成：
- 关键链路可跑通（至少核心冒烟验证通过）。
- 已列出已验证项与未验证项。
- 相关文档入口已更新（根文档或模块文档）。

### 3.6 最低验证命令矩阵

- 后端结构/迁移改动：
  - `go test ./...`
- 前端页面/交互改动：
  - `pnpm exec vue-tsc --noEmit`
- 同时涉及前后端链路：
  - `go test ./...`
  - `pnpm exec vue-tsc --noEmit`
- 若本轮未执行构建或启动服务，必须在总结中显式写明“未验证项”。

### 3.7 未验证项记录格式

- 使用固定格式：`未验证：<命令/场景>，原因：<未执行原因>`。
- 不允许只写“未测”或“待验证”，必须带具体命令或场景。

### 3.8 常见反例（禁止）

- 只改前端显示，不改后端真实权限语义。
- 在运行时保留大量历史兼容分支而不迁移收敛。
- 一次改动同时混入“功能新增 + 大规模重构 + 无关清理”且无分批策略。

## 4. 文档分层规则

- 根目录文档：全局规则、边界、执行清单。
- 模块文档：实现细节、接口说明、迁移细则。
- `docs/`：专题说明、演进记录、变更日志。

若实现与本文档冲突：先核对真实代码与模块文档，再决定是修代码还是更新本文件。
