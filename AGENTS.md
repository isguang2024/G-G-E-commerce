# AGENTS.md

## 全局要求

- 使用中文回答。
- 编辑代码时保证文件编码为 UTF-8，尤其是通过 Shell 写入或保存文件时。
- 优先遵循 `PROJECT_FRAMEWORK.md` 中定义的项目主框架与执行清单；该文件定义总体约束，不承载实现细节。
- 前端改动默认遵循 `FRONTEND_GUIDELINE.md`（搜索区、列表操作列、弹窗工具栏、状态展示统一规范）。
- 涉及前端视觉设计、页面排版、交互观感优化时，默认先调用 `frontend-skill`，再进行实现与样式调整。
- 在完成一次代码咨询时，结尾追加简短总结，说明本次处理结果。
- 当本次任务包含大型代码改动时，结尾必须额外追加固定说明。
- `docs/change-log.md` 只记录大修改、重要节点或功能级收口；简单修补、碎片化过程和短期试错不要写入，优先回写到对应专题文档。

## 大型改动收尾格式

满足以下任一条件，视为大型改动：

- 修改超过 3 个文件
- 涉及核心流程、接口、数据结构、状态管理或数据库逻辑
- 包含重构、迁移、架构调整或跨模块联动修改
- 需要额外说明后续兼容性、测试计划或未完成事项

当判定为大型改动时，最终回复必须追加以下两段，保持简洁：

### 本次改动

- 说明本次改了什么
- 说明为什么这样改
- 说明影响范围或已完成的验证

### 下次方向

- 说明建议的下一步工作
- 说明待确认问题、风险点或可继续优化项

## 大型改动日志文件

- 日志文件路径固定为 `docs/change-log.md`
- 当发生大型改动、重要节点或功能级收口时，除最终回复外，还要将摘要追加到该文件末尾
- 追加内容需包含日期、任务概述、“本次改动”和“下次方向”
- 若本次未执行测试，需要在日志中明确写出未验证项
- 若文件不存在，则自动创建，并使用 UTF-8 编码

如果不是大型改动，则只需在结尾追加 1 至 2 句简短总结，不强制使用上述两段格式。
后续再有架构调整，优先更新专题文档，再决定是否补一条里程碑日志，避免再次堆积过程型日志。
