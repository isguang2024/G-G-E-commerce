# 消息系统设计

> 基线日期：2026-03-31。本文只保留当前正式消息模型与落点。

## 当前结论

- 右上角消息面板保留 `通知 / 消息 / 待办` 三栏。
- 完整消息中心使用 `#/workspace/inbox`。
- 消息系统按“消息主表 + 收件箱表”拆分，内容与投递状态分离。
- 平台消息与团队消息共用一套模型，但作用域和权限分开。

## 正式模型

### 消息主表 `messages`

- 负责保存消息内容、发送者、类型、范围、优先级、动作入口和模板快照。
- 发送者支持 `system / platform_user / team_user / service / automation`。
- 消息类型保留 `notice / message / todo`。

### 收件箱表 `message_deliveries`

- 负责保存每个用户的未读、已读、归档、待办状态。
- 每条投递记录保存来源接收组、来源规则和目标项快照。

## 接收范围

- 支持单用户、多用户、单团队全员、指定角色用户、平台管理员、当前团队管理员。
- 第一阶段已落地的发送对象包括：
  - 平台给所有用户
  - 平台给团队管理员
  - 团队管理员给指定团队成员
  - 接收组规则命中

## 模板与发送人

- `message_templates` 负责模板归属与内容模板。
- `message_senders` 负责平台/团队两套发送人配置。
- 发送时保留发送人快照，避免历史消息随配置变动。

## 权限与接口

- 消息管理相关权限以 `message.manage`、`team.message.manage` 为主。
- 用户侧接口围绕收件箱摘要、列表、详情、已读和待办处理。
- 管理侧接口围绕消息列表、创建、发布和撤销。

## 当前落点

- 平台发信页：`#/system/message`
- 团队发信页：`#/team/message`
- 消息中心：`#/workspace/inbox`
- 平台消息模板、发送人、接收组、发送记录已进入系统页面注册中心。
