# 空间与 Host 架构

> 基线日期：2026-03-31。本文只保留当前可执行约束，以及与当前架构直接相关的演进边界。

## 当前结论

- 当前默认运行模式仍是单域可运行，不强制拆多子域。
- 空间是业务与运行时模型，Host 只是可选的部署映射，不反向定义菜单、页面和权限模型。
- 菜单、受管页面、权限、消息都应先按空间语义建模，再决定是否绑定到独立 Host。
- 当前系统已具备空间与 Host 绑定的底座，但默认行为仍以现有单域流程为准。

## 当前稳定边界

### 空间职责

- 空间负责菜单树归属、默认首页、Host 绑定和少量独立页暴露。
- 菜单负责空间内导航入口，不负责空间切换。
- 受管页面负责非菜单直达访问单元，不按空间复制同一组件定义。
- 权限负责准入、可见性和动作控制，不承担空间外壳切换。

### Host 职责

- Host 只决定进入哪个空间外壳，不能承担菜单组织和权限裁剪。
- 不配置子域时，系统必须完整可运行。
- 配置 Host 绑定后，只能改变入口壳层和跳转方式，不能改写菜单、页面和权限语义。

## 当前已落地底座

- 已有 `menu_spaces` 与 `menu_space_host_bindings` 基础模型。
- `menus` 与 `ui_pages` 已补空间相关字段。
- 默认 `default` 菜单空间已补齐。
- 菜单树与页面运行时已按 Host / 显式空间预留解析入口。
- 当前 Host 绑定相关策略字段已支持 `scheme`、`route_prefix`、`auth_mode`、`login_host`、`callback_host`、`cookie_scope_mode`、`cookie_domain`。

这些底座当前只作为可配置能力保留，不强制改变现有单域登录与 Bearer Token 流程。

## 当前运行规则

- 现阶段优先级固定为：空间模型 > 菜单与页面归属 > 权限准入模型 > Host 配置。
- 当前前端仍使用 `hash` 路由：
  - 默认根路径：`https://host/#/route-path`
  - 带路由前缀：`https://host/prefix/#/route-path`
- 同 Host、同协议、无前缀时，继续使用前端路由跳转。
- Host 改变、协议改变或配置了 `route_prefix` 时，统一走整页跳转。

## 允许的演进方向

- 可以继续保留单域模式作为默认部署方式。
- 可以通过配置把特定空间绑定到特定 Host，但不能把多子域写死为唯一方案。
- 可以继续补强空间准入、跨空间跳转和消息目标空间语义，但前提是当前空间模型不被破坏。

## 当前不建议做的事

- 直接拆多个前端项目。
- 直接按功能模块拆多个子域。
- 为每个空间单独复制一套权限体系。
- 为子域跳转提前引入复杂网关逻辑。

## 验收口径

- 单域模式下，默认空间和非默认空间都能按现有菜单/页面主链稳定运行。
- 未配置额外 Host 时，系统行为不能因空间与 Host 底座存在而发生偏移。
- 配置 Host 绑定后，入口跳转、默认首页和空间解析必须与当前菜单空间语义一致。
