# 文档与导航规范

这份文档用于统一仓库内文档怎么放、怎么写、怎么引用。目标是让新来的人能快速找到真相源，也让协作时不会把说明文写成第二套实现。

若本文件与以下文档冲突，以这些真相源为准：`AGENTS.md`、`PROJECT_FRAMEWORK.md`、`FRONTEND_GUIDELINE.md`、`backend/CLAUDE.md`。

## 1. 分层原则

- `README.md`：仓库总入口，回答“先看什么”
- `docs/README.md`：`docs/` 目录入口，回答“文档从哪开始翻”
- `docs/INDEX.md`：文档索引中枢，回答“每份文档负责什么”
- `backend/api/openapi/README.md`：API 契约真入口，回答”接口怎么改”
- `frontend/`：前端实现真入口，回答“页面和调用怎么落”
- `Instructions/`：agent 执行型说明，回答“协作时怎么做”

## 2. 写作原则

- 只保留当前有效信息，历史说明要么删掉，要么明确标注已失效
- 优先写职责、边界、入口和阅读顺序，不堆长篇背景
- 一份事实只放一个真相源，其他地方只链接，不复述
- 生成物、临时修复、一次性说明不进入长期导航主干

## 2.1 文档命名规范

- 优先使用英文小写文件名，单词之间使用 `-` 连接（例如：`permission-system.md`）。
- 阶段/节点执行记录统一使用 `node-<阶段>-<节点>-<主题>.md`（例如：`node-4-4-quick-lookup-index.md`）。
- 目录级入口文档统一使用 `README.md`。
- 避免同义命名（如 `guide.md` 与 `guideline.md` 并存）导致检索歧义。

## 2.2 Markdown 格式规范

- 标题层级连续递进，不跳级（`#` -> `##` -> `###`）。
- 列表优先扁平结构，避免超过两层嵌套。
- 命令、路径、代码标识统一使用反引号包裹。
- 表格用于“路径-职责/规则-说明”类信息，正文避免超长段落。
- 链接优先相对路径，提交前必须校验可跳转。

## 3. 链接原则

- 站内优先用相对路径
- 入口页优先指向唯一真相源，而不是多份并列说明
- 专题文档优先从 `docs/guides/README.md` 进入，再去看具体条目
- 当一个主题已有专门入口时，新文档不要重复造同义页

## 4. 协作约定

- API 相关工作先看 `backend/api/openapi/README.md`
- 前端实现先看 `FRONTEND_GUIDELINE.md`
- 当前结构和边界优先参考 `PROJECT_FRAMEWORK.md` 与 `docs/project-structure.md`
- `Instructions/` 如果存在，只放 agent 级别的执行约束，不放业务背景文

## 4.1 文档位置指南

- 放在 `docs/`：面向团队的长期说明、流程、规范、任务导航与执行记录。
- 放在 `.claude/`：协作工具本地配置、技能定义、会话辅助信息。
- 放在 `backend/` 或 `frontend/` 子目录：与该代码域强绑定的实现说明。
- 若同一主题已在真相源定义，新增文档仅补导航，不重复解释。

## 5. 新文档落点

新增文档先回答三个问题：

1. 它是总入口、专题手册，还是阶段任务？
2. 它是否会长期维护？
3. 它是否在索引中需要被显式点名？

落点建议：

- 总入口类内容放 `README.md` 或 `docs/README.md`
- 导航地图放 `docs/INDEX.md`
- 规范和约束放 `docs/GUIDELINES.md`
- 具体做法放 `docs/guides/`

## 6. 维护规则

- 改导航前先确认路径是否真实存在
- 改入口后同步检查相关引用是否还指向正确文件
- 新增专题文档时，优先补 `docs/INDEX.md` 的索引条目
- 如果某份文档已经不再代表当前事实，直接删链路，不保留模糊措辞

## 7. 文档更新流程

1. 明确文档类型（入口 / 规范 / 指南 / 执行记录）。
2. 在对应目录创建或更新文档，并遵循命名与格式规范。
3. 若新增长期文档，同步更新 `docs/INDEX.md`。
4. 若替换旧文档，先修复引用再删除旧文件。
5. 在任务收口时记录变更证据（文件路径、关键改动、校验结果）。
