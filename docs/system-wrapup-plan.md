# 系统部分收尾计划

> 更新时间：2026-03-30。目标不是继续扩系统页，而是把现有系统底座收稳、收紧、可验收，为后续业务模块承载做准备。

## 1. 当前判断

当前系统侧已经具备可持续承载业务的正式主链：

- 权限、功能包、快照主链已稳定
- 菜单与页面正式分层已稳定
- 菜单空间与可配置 Host 底座已落地，但仍以单域兼容运行
- 消息系统已从 mock 进入真实发送、收件、记录链路

当前最缺的不是新能力，而是：

- 系统回归基线
- 临时兼容与历史残留收紧
- 空间化启用前的前置闭环
- 演示数据与验收场景

## 2. 收尾目标

本轮系统收尾只聚焦四件事：

1. 建立系统回归基线，避免后续业务开发频繁打回菜单、页面、消息、空间链路
2. 收紧明显的临时提示、重复入口和过期兼容分支
3. 把空间化保持在“默认单域可运行、按需多空间/多 Host 可启用”的稳定状态
4. 为后续业务模块进入做好最小演示与验收条件

## 3. 执行计划

### 3.1 第一阶段：稳定性基线

- 建立系统验收清单，覆盖菜单、页面、菜单空间/Host、消息系统四条链
- 明确每条链的最小验证动作、预期结果、依赖账号与数据
- 后续每次改系统链路，都优先按这份清单做回归
- 固定清单见 [system-regression-checklist.md](/C:/Users/Administrator/Documents/GitHub/G-G-E-commerce/docs/system-regression-checklist.md)
- 固定演示数据见 [system-demo-data.md](/C:/Users/Administrator/Documents/GitHub/G-G-E-commerce/docs/system-demo-data.md)

### 3.2 第二阶段：临时逻辑收紧

- 去掉明显依赖弹窗提示的临时降级逻辑，改成静默收敛或界面内状态说明
- 清理系统管理页里低价值重复入口、重复文案和历史兼容提示
- 对仍需保留的兼容分支，明确“为什么保留、未来何时可以继续收缩”
- 当前兼容审计见 [system-compatibility-audit.md](/C:/Users/Administrator/Documents/GitHub/G-G-E-commerce/docs/system-compatibility-audit.md)

### 3.3 第三阶段：空间化进入闭环

- 保持默认 `default` 菜单空间继续兼容当前单域单菜单运行
- 补齐空间准入、默认首页、Host 绑定、跨空间导航的验收链路
- 真正启用多 Host 前，只继续增强配置与校验，不急着切部署模式

### 3.4 第四阶段：演示与交付条件

- 补平台管理员、团队管理员、普通成员三类最小演示账号与团队数据
- 补至少一个非默认菜单空间和一组 Host 绑定示例
- 补消息模板、发送人、接收组、发送记录的最小演示数据
- 初始化命令固定为 `go run ./cmd/init-demo`
- 当前演示策略统一为：
  - 团队管理员承担团队消息与团队空间链路回归
  - 普通成员承担菜单裁剪、页面准入与平台消息收件回归

## 4. 系统验收主清单

### 4.1 菜单管理

- 菜单管理按当前菜单空间正确过滤，不串空间
- 菜单分组、菜单挂接页面、菜单备份主链可打开、可保存、可刷新
- 管理模式树与运行时菜单树的空间过滤口径一致

### 4.2 页面管理

- 页面列表、页面候选、挂接候选、父链候选按 `space_key` 正常过滤
- 独立页面、挂接页面、逻辑分组、普通分组四类页面均可创建与编辑
- 页面“访问”入口与运行时路由保持一致

### 4.3 菜单空间 / Host

- 默认空间在未配置额外 Host 时继续单域可运行
- 非默认空间可创建、可初始化、可重新初始化、可查看状态
- 默认首页保存前后都能校验
- Host 绑定冲突、主绑定唯一、禁用空间不可启用 Host 等规则正常生效

### 4.4 消息系统

- 消息发送、消息中心、发送记录、模板、发送人、接收组主链可用
- 异步发送状态可观察
- 消息正文中的站内链接在消息中心和发送记录详情里都可跳转
- 平台消息与团队消息入口继续分开，但交互结构一致

## 5. 当前不建议优先做的事

- 不建议立即启用真实多子域部署
- 不建议继续在系统管理里扩大量新页面
- 不建议再引入新的平行权限、菜单、页面模型
- 不建议在未建立演示数据前继续深推团队侧复杂规则

### 5.1 历史副本目录暂不直接删除

当前仓库未检索到对 `frontend-copy` 的代码引用，它更像历史副本目录。  
但本轮系统收尾不直接删除，原因是：

- 仅靠代码搜索不足以证明它对人工备份流程完全无用
- 删除属于高风险收缩动作，不适合在系统收尾阶段顺手执行

建议处理方式：

- 先保留为“疑似历史副本”状态
- 等确认不再承担人工备份用途后，再单独清理
- 审计记录见 [system-compatibility-audit.md](/C:/Users/Administrator/Documents/GitHub/G-G-E-commerce/docs/system-compatibility-audit.md)

## 6. 下一阶段建议

- 系统底座稳定后，尽快用真实业务模块进入压测，而不是继续长期停留在系统页优化
- 优先选择一个真正依赖权限、菜单、页面、消息的业务模块验证当前架构
- 若业务模块进入后暴露新问题，再回到系统层补底座，不做脱离业务的过度系统化
