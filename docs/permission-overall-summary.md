# 权限系统总体总结

> 文档收敛说明（2026-03-25）：历史重复设计文档 `permission-package-design.md` 已下线，本文件作为权限体系单一总览文档继续维护。

## 1. 当前方向
当前权限系统已经正式从“零散权限直配”切到“功能包优先”。

一句话总结：
- 功能包决定有没有模块。
- 展开后的菜单和 `permission_key` 决定模块内能看到什么、能做什么。

## 2. 已固定的核心原则
### 2.1 上下文
系统只保留两个上下文：
- `platform`
- `team`

默认进入规则：
- 有平台能力时，默认进入 `platform`
- 无平台能力但有团队时，默认进入 `team`
- 仅 `team` 上下文发送 `X-Tenant-ID`

### 2.2 功能包优先
功能包是唯一业务开通单元。

它决定：
- 是否拥有模块
- 菜单候选范围
- 权限候选范围

### 2.3 权限键职责收敛
`permission_key` 继续保留，但只负责：
- 自动注册
- 运行时最终鉴权
- 作为功能包绑定的原子能力项

补充规则：
- 正式权限键统一使用点式格式。
- 历史 `resource:action` 仅保留兼容输入语义，运行时快照、接口输出和前端判断都应归一化为点式 key。
- 权限自动注册时必须显式写入 `context_type`；不能依赖 `permission_actions` 模型默认值 `team`，否则平台系统权限会被错误写入团队上下文。

### 2.4 菜单统一管理
- 菜单定义在全量菜单页维护
- 功能包只绑定菜单
- 公共菜单不依赖功能包

### 2.5 角色和用户只做减法
- 平台角色：可裁剪包内菜单、权限
- 平台用户：可裁剪菜单
- 团队边界：只做减法
- 团队角色：只做减法
- 团队成员个人例外：已退出主链

## 3. 当前正式模型
### 3.1 功能包模型
功能包分为：
- 基础包
- 组合包

规则：
- 基础包直接绑定菜单和权限
- 组合包只包含基础包
- 组合包不直接绑定菜单和权限

### 3.2 平台侧
平台最终能力来源于：
- 全局角色包并集
- 用户直绑包
- 角色减法裁剪
- 用户菜单减法裁剪

### 3.3 团队侧
团队最终能力来源于：
- 团队已开通包
- 团队边界减法
- 团队角色减法

团队授予链补充规则：
- 团队和团队角色都可以使用 `team` 与 `platform,team` 功能包。
- 纯 `platform` 功能包不能进入团队侧。
- 基础团队角色的功能包查看页需要展示当前团队全部可用功能包，并只读标识哪些包已被继承，避免把“看不全”误判成缺包。

团队成员个人不再作为正式授权层。

## 4. 当前实现状态
已经落地：
- 平台优先上下文
- 平台用户菜单裁剪主链
- 平台用户权限例外降为只读审计
- 平台角色快照读链
- 团队边界菜单/权限减法
- 团队角色减法交互
- 默认内置包与默认角色开包初始化
- 菜单来源页与摘要组件统一

尚未完整落地：
- 组合包运营视图与来源预览仍需继续增强
- 权限系统的“开发就绪清单”还缺最后一轮整理

### 4.1 读链状态
当前读链收口为：
- 平台用户：平台用户快照优先
- 平台角色：平台角色快照优先
- 团队边界：团队快照优先
- 团队角色：团队边界服务/角色快照优先
- 平台鉴权：`authorization` 在平台上下文下已改为快照强主链，不再按 `HasPackageConfig` 回退旧正向权限表。
- 团队鉴权：`authorization` 在团队上下文下已改为角色快照主链，不再回退团队成员个人例外和旧角色正向权限表。
- 快照缺失时：平台用户、平台角色、团队边界、团队角色均已改为“缺失即刷新并持久化”，减少运行时临时现算结果。

### 4.2 兼容层状态
当前仍保留但已降级的旧链：
- `user_action_permissions`

它们不再是主链，只承担兼容审计或兜底职责。

补充：
- `tenant_action_permissions`、`team_manual_action_permissions`、`role_menus`、`role_action_permissions` 已物理删除。
- 平台角色侧旧正向直配回退已退出主链；当前平台角色菜单与权限边界统一走角色快照。
- 平台角色页面的 `has_*` 边界兜底判断已退出主链，前端统一按快照返回的可用集合过滤展示。

## 5. 快照策略
当前正式性能方向：
- 以数据库快照表为第一阶段主方案
- 读取优先命中快照
- 写入后统一刷新

当前核心风险：
- 不是规则定义错误
- 而是刷新触发不全

因此已经把刷新机制提升为正式规范，并明确触发源：
- 功能包绑定关系变更
- 组合包包含关系变更
- 菜单启停变更
- 角色绑定包变更
- 用户直绑包变更
- 角色减法裁剪变更
- 用户减法裁剪变更
- 团队开包变更
- 团队边界减法变更
- 团队角色裁剪变更
- 成员角色变更

## 6. 当前最重要的收口方向
### 第一优先级
- 继续把运行时权限读链切到快照
- 进一步降低旧正向表在鉴权中的参与度

### 第二优先级
- 继续统一平台用户、平台角色、团队边界、团队角色四类页面的来源视图、空态、反馈和跳转
- 保持“展开结果 / 显式屏蔽 / 最终结果”完全同构

### 第三优先级
- 持续下线旧兼容写链
- 只保留最小兼容读能力

## 6.1 当前仍缺的功能
### P0: 功能包集合（组合包）能力补齐
- 已完成：
  - `featurepackage.Service/Handler` 已补齐“查询子包 / 保存子包关系”正式接口
  - 功能包管理页已拆成“基础包 / 组合包”双页签
  - 创建/编辑弹窗已支持 `package_type`
  - 组合包已支持独立“配置基础包”交互
- 当前剩余缺口：
  - 角色/团队授予页虽然已可看到组合包展开出的基础包、权限、菜单预览，但还缺更强的按包钻取与明细检索
  - 组合包来源展示仍以授予页预览和来源标签为主，尚未形成独立运营视图

### P0: 组合包约束校验
- 已完成部分：
  - 已校验自包含、bundle 仅能包含 base、上下文兼容、bundle/base 切换前置条件
  - 删除功能包时已清理父子关系并触发父包刷新
- 仍待补齐：
  - 循环依赖显式校验
  - 停用组合包/基础包后的影响范围反馈

### P1: 功能包运营视图
- 已完成：
  - 功能包列表已支持 `package_type` 双页签、展示和统计拆分
  - 默认内置组合包 `platform.admin_bundle` 和管理员默认授予已初始化落地
  - 角色/团队授予页已能直接看到组合包展开后的基础包、权限、菜单预览
- 仍待补齐：
  - 功能包详情级的组合来源预览仍偏轻量
  - 缺少面向运营的组合包影响范围总览与搜索能力

### P1: 开发期辅助能力
- 缺少一份“权限系统开发就绪清单”，明确新增模块时要不要新建基础包、要不要纳入组合包、要不要刷新哪些主体。
- 缺少组合包相关测试：递归展开、去重、上下文过滤、停用过滤、刷新联动都应有最小自动化覆盖。

## 7. 当前结论
这套权限系统当前已经不是“按 API 配权限”的模型，而是：
- 以功能包作为业务开通入口
- 以菜单和 `permission_key` 作为展开结果
- 以减法裁剪作为角色和边界控制方式
- 以快照作为运行时主读链方向

后续所有实现和迁移，都应围绕这条主线继续推进。

## 当前补充结论
- 功能包上下文需要保留，但应支持双上下文功能包，而不是把所有包强制拆成两份。
- 当前重复团队菜单属于旧菜单迁移残留，不是前端重复渲染；迁移脚本已经开始清理旧 `/team` 根树。
- 当前冒号格式权限键属于旧回退链遗留；迁移与映射已改为优先输出点式 key。

## 2026-03-24 来源页统一补充
- 团队边界来源接口已正式以 `blocked_*` 作为主字段，前端主链不再消费 `manual_*` 历史别名。
- 平台角色、团队角色、团队边界、平台用户四类来源页继续统一为同一套交互：功能包展开结果、显式屏蔽/关闭结果、最终保留结果。
- 共享来源面板已统一“全部功能包 / 当前筛选下暂无展开项 / 当前筛选下暂无显式裁剪项 / 当前暂无来源明细”这组反馈文案，后续继续减少页面分叉。

## 2026-03-24 上下文与权限键收口补充
- 权限自动注册、权限列表接口输出、前端权限归一化三条链已统一补齐上下文推导，平台系统权限不再因为缺失 `context_type` 而默认落入 `team`。
- `permission_actions` 的点式 `permission_key` 已进一步收口：列表输出与前端展示优先返回点式 key，历史冒号格式仅保留兼容输入，不再作为正式展示值。
- 当前改版进度更新：
  - 规则与模型：约 `96%`
  - 后端主链：约 `91%`
  - 前端交互与心智统一：约 `86%`
  - 迁移与数据一致性：约 `94%`

## 2026-03-24 最终缺口判断
- 功能包集合不是“完全没做”，而是“底座已做，产品能力未闭环”。
- 已有部分：
  - 数据表与模型：`feature_package_bundles`
  - 运行时展开：平台、团队、角色、用户读取链都会递归展开 bundle
  - 刷新联动：`RefreshByPackage` 已考虑子包到父包的反查刷新
- 还缺关键闭环：
  - 循环依赖与停用影响的更强校验反馈
  - 组合包详情级运营视图与影响范围总览

## 2026-03-24 权限主数据收口决策
- 正式主链收口为：`API -> 功能键 -> 功能包 -> 授权对象`。
- `api_endpoints` 与 `api_endpoint_permission_bindings` 继续保留，作为 API 自动注册与 API 归属功能键的正式模型。
- `feature_packages`、`feature_package_bundles`、`role_feature_packages`、`user_feature_packages`、`team_feature_packages` 继续保留，作为功能包与授权对象的正式模型。
- `permission_actions` 当前同时混合“功能键主数据 / 动作语义 / API 关联”三层职责，不再继续作为长期正式模型扩张；后续应迁移收口为单独的 `permission_keys` 主表，`feature_package_actions` 也同步演进为 `feature_package_keys`。

## 2026-03-24 API 元数据策略决策
- API 元数据采用“代码注册时内联声明”的正式策略，而不是外部配置文件全量驱动。
- 原因是 API 元数据与真实路由强耦合，跟随注册点声明最不容易漏，且最符合“项目里的 API 自动注册”目标。
- 但初始化默认数据仍应独立为 Seeds 模块，专门维护基础功能键、管理员功能包、默认角色绑定和固定 UUID。
- 后续正式分层为：
  - 路由注册阶段：注册 API，并同步 `method/path/module/summary/permission_keys/category/source`。
  - 初始化阶段：导入默认功能键、默认功能包、默认角色绑定等主数据。
- 自动注册范围进一步收口为：只有“带元数据的 API”才会自动进入 API 单元；没有元数据的普通 API 不自动写入 `api_endpoints`。
- 对于未自动注册的普通 API，系统应允许管理员在 API 管理页面手动创建 API 单元，并绑定到实际路由。
- API 管理页必须提供“搜索未注册 API”能力，用于从当前运行中的真实路由里筛出尚未进入 API 单元的接口。

## 2026-03-24 API 元数据注册器方向
- 现有 `internal/pkg/apiregistry` 已经是路由元数据唯一入口，后续应继续在该包内扩展链式 Builder，而不是把元数据逻辑散落到业务模块。
- 正式目标是提供类似“类方法”式的注册体验，减少每个 API 重复书写，例如：
  - `Meta("获取用户列表").绑定分组("user").绑定来源("sync").绑定功能键("system.user.manage").Build()`
- `Registrar` 继续负责 `GET/POST/PUT/DELETE` 注册；Builder 只负责组织 `RouteMeta`，以兼容现有 `*RouteMeta` 调用方式并允许渐进迁移。
- 该 Builder 还应内置统一规范：权限键去重、模块默认继承、分类默认推导、稳定 endpoint code 推导。
- API 元数据正式字段已收口为：
  - `code`
  - `module`
  - `summary`
  - `feature_kind`
  - `category_code`
  - `context_scope`
  - `source`
  - `permission_keys`
- 历史 `resource_code / action_code` 不再属于 `RouteMeta` 正式模型，也不应再通过 Builder 暴露；权限归属统一改由 `permission_key` 表达。

## 2026-03-24 功能权限旧列清理补充
- `permission_actions.source`、`permission_actions.resource_code`、`permission_actions.action_code` 已从正式模型、迁移链路和数据库中移除。
- 功能权限正式主链固定为：
  - `permission_key`
  - `module_code`
  - `module_group_id`
  - `feature_group_id`
  - `context_type`
- 前端权限页、功能包权限选择、API 绑定展示都不应再依赖旧 `resource/action` 二段式字段；若需要展示兼容信息，只能从点式 `permission_key` 派生只读显示值。

## 2026-03-24 部署初始化 Builder 决策
- 部署链路应额外提供独立的初始化 Builder，用于串联默认分类、默认功能键、默认功能包、默认角色绑定和带元数据 API 的同步。
- 该 Builder 的职责是“部署与初始化”，不是“业务路由注册”；因此不应把初始化逻辑塞回各模块 `module.go`。
- 默认主数据应继续从 `internal/pkg/permissionseed` 单点输出，至少包括：
  - 默认 API 分类
  - 默认功能键
  - 默认功能包
  - 功能包菜单绑定
  - 功能包功能键绑定
  - 组合包包含关系
  - 默认角色功能包绑定
- `cmd/migrate/main.go` 的职责应收口为“消费 Seeds 并写库”，不再继续维护第二套本地默认数组。
- 初始化 Builder 需要具备基础可观测性，至少能输出：
  - 默认分类数量
  - 默认功能键数量
  - 默认功能包数量
  - 默认组合包关系数量
  - 默认角色绑定数量
  - 带元数据可自动注册的路由数量
  - 未注册路由数量

## 2026-03-24 初始化执行层收口补充
- `internal/pkg/permissionseed` 已不再只是“静态 Seeds 常量目录”，同时承载默认主数据的执行型 Ensure 能力。
- 当前已由 `permissionseed` 统一负责的初始化动作包括：
  - 默认 API 分类
  - 默认权限分组
  - 默认功能键
  - 默认功能包
  - 默认组合包关系
  - 默认角色功能包绑定
- `cmd/migrate/main.go` 的职责进一步收口为：
  - 组织执行顺序
  - 记录日志与摘要
  - 串联刷新与补偿逻辑
- 后续新增默认主数据时，应优先补到 `permissionseed`，而不是把新数组和写库逻辑直接塞回迁移入口。

## 2026-03-24 API 管理页补录闭环补充
- API 管理页现阶段必须具备“未注册 API”弹窗能力，可直接查询运行时未注册路由。
- 查询维度至少包括：
  - Method
  - 路径
  - 模块
  - 关键字
  - 是否仅查看无元数据路由
- 管理员从未注册路由发起“创建 API”时，页面应自动把已有路由信息带入新增 API 表单，包括：
  - method
  - path
  - module
  - summary
  - feature_kind
  - category_code 对应分类
  - context_scope
  - source
  - permission_keys
- 该补录链路以“简单稳定”为第一优先级，当前不要求在未注册弹窗内直接完成复杂编辑。
- API 管理页还应提供基础状态概览，至少能看见：
  - 自动注册 API 数量
  - 手工补录 API 数量
  - 初始种子 API 数量
  - 未注册路由数量
- API 主列表应支持按注册方式快速筛选，至少包括：
  - `sync`
  - `manual`
  - `seed`
- API 管理页还应提供统一查询条，至少支持：
  - Method
  - 路径
  - 分类
  - 团队上下文
  - 状态

## 2026-03-24 删除联动与快照一致性补充
- 权限系统中的删除操作，不应只依赖单表删除；正式规则应为“业务服务编排删除 + 纯从属表允许级联删除”的混合模式。
- 适合由服务层显式编排删除的对象包括：
  - 角色删除
  - 团队删除
  - 功能包删除
  - API 单元删除
- 这些删除除去主体本身，还必须同步处理：
  - 绑定关系表
  - 显式拒绝/隐藏表
  - 快照表
  - 删除后的刷新动作
- 适合数据库外键 `ON DELETE CASCADE` 的，只应是“纯从属、无业务分支”的表，例如典型的关联中间表或缓存快照表；只要删除永远正确，就可以下推到数据库层。
- 只要删除动作会影响权限快照，就必须在事务提交后刷新相关快照，不能仅删除记录后依赖懒加载空返回。
- 当前正式约束：
  - 平台角色删除后，必须删除角色绑定的功能包、菜单隐藏、功能禁用、数据权限、用户角色绑定、平台角色快照，并刷新受影响的平台用户快照。
  - 团队角色删除后，必须删除该团队下该角色的全部成员绑定、角色功能包、角色菜单隐藏、角色功能禁用、角色数据权限、团队角色快照，并刷新团队快照。
  - 团队删除后，必须同时清理团队功能包、团队边界阻断、团队角色、团队成员、团队用户例外权限、团队快照和团队角色快照，避免残留孤儿权限数据。
- 快照读取规则同步收口为：
  - 若快照存在，则直接返回。
  - 若快照缺失，则自动重新计算并持久化，再返回结果。
  - 不允许再以“返回空快照”作为缺失快照的默认处理。

## 2026-03-24 权限主表命名收口补充
- 数据库正式主表名已收口为：
  - `permission_keys`
  - `feature_package_keys`
- 历史表名：
  - `permission_actions`
  - `feature_package_actions`
  现在仅作为迁移兼容来源，不再是正式模型名。
- 迁移入口会在 `AutoMigrate` 前优先执行旧表重命名，保证老环境升级时不会先创建一套新空表。
- 当前兼容策略为：
  - 数据库表名已切换到新命名
  - Go 结构名、Repo 名、Service 名现已同步收口到 `PermissionKey / FeaturePackageKey`
  - 仓储、鉴权、初始化、索引维护均已改为落新表
- 当前仍保留兼容的主要范围只包括：
  - API 路径中的 `/actions`
  - JSON 字段中的 `action_ids / action_id`
  - 少量历史表名对应的仓储实现名，例如 `RoleDisabledActionRepository`
- 这些兼容命名只用于降低外部破坏面，不再代表系统内部正式模型仍是旧 action 语义。

## 2026-03-24 默认权限分组固定化补充
- 功能权限的默认分组不允许再通过“扫描已有权限键 code 自动生成”得到。
- 当前正式规则改为：
  - 默认功能分组走固定 Seed 清单
  - 默认模块分组走固定 Seed 清单
  - 所有默认分组 ID 使用稳定 UUID 固定生成
  - 部署迁移时统一由 `permissionseed` 导入
- 当前内置功能权限默认绑定策略：
  - 默认功能分组统一绑定到 `system`
  - 默认模块分组按现有系统模块显式绑定
- 当前默认模块分组最少应覆盖：
  - `role`
  - `permission_key`
  - `user`
  - `menu`
  - `menu_backup`
  - `system`
  - `tenant`
  - `tenant_member_admin`
  - `team_member`
  - `team`
  - `api_endpoint`
  - `feature_package`
  - `system_permission`
- 这些默认模块分组必须作为“可迁移主数据”存在，不能依赖运行时再推导名称。

## 2026-03-24 permission_action 到 permission_key 正式迁移补充
- `permission_action` 现在只允许作为历史兼容映射存在，不再允许作为正式模块编码、API 分类编码或默认模块分组编码继续落库。
- 当前正式编码统一为：
  - API 分类：`permission_key`
  - 默认模块分组：`permission_key`
  - 内置功能键 `module_code`：`permission_key`
- 迁移链路必须满足以下顺序：
  - 先确保 `permission_key` 对应的默认 API 分类、默认模块分组和默认功能键已存在
  - 再把旧 `permission_action` 关联的数据引用迁到 `permission_key`
  - 最后删除旧 `permission_action` 分类与旧模块分组
- 这条迁移必须保持幂等，允许在老环境重复执行，直到数据库中只剩 `permission_key` 正式编码为止。
