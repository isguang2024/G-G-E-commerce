# 权限改造总体总结

## 1. 这次整体改了什么
本轮权限改造的核心，不是修补原有 API 动作权限，而是把整套模型从“接口判权体系”重构成“业务能力体系”。

已经完成的主线变化：
- 从 `resource_code + action_code + scope` 收敛到 `permission_key`。
- 从“API 是权限主体”收敛到“API 只是功能权限的落点”。
- 从“团队边界直接配一堆权限”升级到“功能包开通团队，团队再做细分分配”。
- 从“全局角色混用团队角色”升级到“平台角色 / 基础团队角色 / 团队自定义角色”分层。

## 2. 已完成的关键改造

### 2.1 权限主键统一为 `permission_key`
已经完成：
- 后端鉴权改为以 `permission_key` 为主。
- 前端菜单、按钮、页面权限统一消费 `permission_key`。
- CRUD 风格旧权限串已经退出业务主线。
- `scope` 已从权限主链移除。
- `category` 已删除。

结果：
- 权限模型不再跟 API 粒度强耦合。
- 一个业务权限可以稳定覆盖多个接口、多个页面动作。

### 2.2 上下文从“作用域”改为“平台/团队”
已经完成：
- 引入 `context_type`。
- 功能权限支持 `platform/team`。
- 功能包支持 `platform/team`。
- 产品规则明确为“系统管理偏平台，业务能力偏团队”。

结果：
- 后续新模块不会再围绕 `scope` 扩展。
- 以后增加商品系列、渠道系列、内容系列时，可以直接判断它属于平台还是团队上下文。

### 2.3 API 注册退回到元数据层
已经完成：
- API 注册表继续保留。
- API 现在归属到 `permission_key`。
- 页面展示已补“接口规格”和“鉴权模式”。
- API 管理支持按模块分类查看。

结果：
- API 仍可用于审计、说明、同步。
- 但 API 已不再是权限设计起点。

### 2.4 功能包模型已正式落地
已经完成：
- 落库 `feature_packages`
- 落库 `feature_package_actions`
- 落库 `team_feature_packages`
- 落库 `team_manual_action_permissions`
- 功能包 CRUD 已有。
- 功能包配置包含权限已接通。
- 团队开通功能包已接通。
- 功能包页可反查已开通团队。

结果：
- 平台可以按“能力产品”而不是按零散权限给团队开通业务。
- 团队内部再细分角色/个人权限时，有了正式上游来源。

### 2.5 团队边界模型已经拆层
已经完成：
- 功能包展开权限与团队手工补充权限分开存储。
- 团队权限页已展示来源摘要。
- 团队成员个人权限覆盖只允许落在团队已开通能力内。

当前三层：
- 功能包展开来源
- 团队手工补充
- `tenant_action_permissions` 最终生效边界

结果：
- 团队边界不再只能靠一张表硬扛所有语义。

### 2.6 团队自定义角色已落地
已经完成：
- `roles.tenant_id` 已落库并迁移。
- 平台角色和团队角色查询已分开。
- 平台侧系统角色接口不再直接维护团队自定义角色。
- 当前团队角色 CRUD 已上线。
- 当前团队角色菜单权限维护已上线。
- 当前团队角色功能权限维护已上线。
- 团队成员分配角色已接当前团队角色集合。

结果：
- 团队内部已经具备真正的角色管理能力，而不是继续依赖平台全局角色池。

## 3. 当前正式权限模型
当前模型可以概括成四层：

1. 功能模块
2. 功能权限 `permission_key`
3. 功能包
4. API

固定关系：
- 功能模块 -> 多个功能权限
- 功能权限 -> 多个 API
- 功能包 -> 多个功能权限

团队使用链路：
- 平台定义功能权限
- 平台定义功能包
- 平台给团队开通功能包
- 团队基于已开通能力给角色和成员分配

## 4. 这套模型现在的优点
- 不再被接口数量绑架，权限粒度更稳定。
- 适合后续扩展更多业务模块，不必每次重复设计。
- 支持平台管理与团队业务分层。
- 支持“平台开通能力，团队内部再细分”的组织方式。
- 已经能满足未来大部分“多模块、团队可授权、平台可控”的场景。

## 5. 当前仍未彻底完成的点

### 5.1 `tenant_action_permissions` 还没有彻底降级成纯缓存
当前它已经不再是唯一来源，但运行时判定和部分管理链路仍直接依赖它。

目标应是：
- 功能包展开 + 团队手工补充 = 真正来源
- `tenant_action_permissions` 只负责缓存最终可用集合

### 5.2 团队角色与功能包边界的交互还可继续强化
当前已经能配，但还可以更直观：
- 在团队角色功能权限页标明权限来源模块。
- 明确哪些能力来自功能包、哪些是团队补充。
- 把前端交互彻底收成“仅能选当前团队已开通能力”。

### 5.3 团队成员侧仍有少量体验收尾项
例如：
- 区分基础角色与团队自定义角色。
- 继续弱化旧兼容概念在页面里的存在感。
- 进一步统一成员权限页的说明文案。

### 5.4 数据权限层尚未并入这一轮主线
如果未来要做更细的数据访问边界：
- 需要单独规划团队角色/团队成员的数据权限模型。
- 不建议和当前功能权限主线混做。

## 6. 下一步建议

### 第一优先级
- 把 `tenant_action_permissions` 彻底降级为缓存。
- 增加统一的团队边界刷新服务，避免各链路分别维护最终生效权限。

### 第二优先级
- 继续强化团队角色和团队成员分配页面，只允许选择团队已开通能力。
- 在团队角色/成员权限页显式展示功能包来源和团队补充来源。

### 第三优先级
- 给功能包增加更完整的平台管理视图：
  - 包含权限数
  - 已开通团队数
  - 模块分布
  - 使用范围

### 第四优先级
- 如果后续开始做新业务模块，按现在的规则直接接入：
  1. 定义 `permission_key`
  2. 定义 `context_type`
  3. 归入功能包
  4. 再接 API 和菜单

## 7. 功能包页面是否需要独立管理
需要独立管理页，不建议直接并到功能权限页里。

原因：
- 功能权限页负责定义“最小能力单元”，这是权限模型底层。
- 功能包页负责管理“能力组合、团队开通、使用范围”，这是能力产品层。
- 功能包后续天然会继续增长出团队数统计、模块覆盖、启停、授权范围、开通团队等管理动作，这些都不属于功能权限定义本身。
- 如果把功能包直接塞进功能权限页，短期看入口少了，长期会让一个页面同时承担“权限定义”和“能力分发”两类职责，后面会再次变重。

推荐结构：
- 功能权限页：只管权限定义、上下文、来源、模块归属。
- 功能包页：只管组合权限、开通团队、查看使用范围和统计。
- 两个页面之间保持跳转联动，而不是合并成一个页面。

## 8. 总体结论
当前这套权限改造已经完成了最难的结构性部分：
- 从 API 权限体系切换到业务功能权限体系。
- 从单层团队边界切换到功能包驱动的团队能力模型。
- 从全局角色混用切换到平台角色与团队角色分层。

下一步不需要再大改方向，重点是继续把现有模型收紧、做透、做稳。后续新业务模块可以直接在这套结构上接，不需要再重新设计一套权限系统。
