# 权限系统总体设计总结

## 1. 当前总结
当前权限系统的正式方向已经从“零散权限配置”切换为“功能包优先”。

这意味着：
- 业务开通看功能包。
- 菜单显示候选范围看功能包。
- 权限候选范围看功能包。
- 后端最终仍按 `permission_key` 校验。
- 角色和用户只在已生效功能包范围内做减法裁剪。

一句话：

`功能包决定有没有模块`

`最终展开权限决定模块里能做什么`

## 2. 已经确认的核心原则

### 2.1 上下文
系统只保留两个上下文：

- `platform`
- `team`

所有非团队业务，统一归 `platform`。

### 2.2 功能包是唯一业务开通单元
不再以零散 `permission_key` 作为业务开通入口。

功能包现在是：
- 模块边界
- 菜单边界
- 权限边界

### 2.3 权限键仍然保留
`permission_key` 仍然存在，但职责已收敛为：
- 原子权限定义
- 自动注册对象
- 后端最终鉴权键
- 功能包绑定对象

### 2.4 菜单仍全量维护
菜单定义继续在全量菜单页维护。

功能包不创建菜单，只绑定菜单。

### 2.5 组合能力正式支持
功能包正式拆成：
- 基础包
- 组合包

基础包绑定菜单和权限，组合包只包含基础包。

## 3. 当前正式对象关系

### 3.1 原子层
- 菜单
- `permission_key`

### 3.2 功能包层
- 基础包
- 组合包

### 3.3 授予层
- 平台侧：全局角色、全局用户
- 团队侧：团队、团队角色

### 3.4 运行时层
- 最终菜单集合
- 最终权限集合

## 4. 各层职责已经固定

### 4.1 功能包
负责：
- 决定是否拥有模块
- 决定菜单候选范围
- 决定权限候选范围

不负责：
- 直接做后端最终鉴权
- 直接维护 API
- 直接维护按钮动作

### 4.2 菜单
负责：
- 页面入口
- 导航结构

特殊规则：
- 菜单可被多个功能包复用
- 菜单停用优先级高于功能包绑定
- 支持公共菜单

### 4.3 权限键
负责：
- 最终运行时鉴权
- 按钮/操作能力判断

### 4.4 角色和用户
负责：
- 在已生效功能包范围内做减法裁剪

## 5. platform 正式规则

### 5.1 包来源
平台最终包：

- 全局角色功能包并集
- 加上全局用户直绑功能包

### 5.2 裁剪规则
平台允许用户级减法裁剪：

- 用户隐藏菜单

平台角色也允许统一做减法裁剪：

- 角色隐藏菜单
- 角色关闭权限

### 5.3 最终效果
平台菜单和权限都从平台最终包展开，再经过角色和用户减法裁剪。

## 6. team 正式规则

### 6.1 团队开包是上限
团队已开通功能包决定团队上下文最大能力边界。

### 6.2 团队层只做减法
团队边界正式保留，但只做减法。

拆分为：
- 团队屏蔽菜单
- 团队屏蔽权限

### 6.3 团队角色
基础团队角色：
- 固定内置
- 默认继承团队全部可用包
- 不可删，不可改编码

团队自定义角色：
- 默认继承团队全部可用包
- 允许裁剪成团队包子集
- 不允许拿团队未开通包

### 6.4 团队成员个人层
团队成员个人例外正式取消。

团队内能力只通过：
- 团队开包
- 团队边界减法
- 团队角色裁剪

来决定。

## 7. 菜单最终规则
菜单最终显示逻辑固定为：

1. 先看当前上下文已生效功能包
2. 展开出菜单候选范围
3. 过滤停用菜单
4. 叠加角色/用户减法裁剪
5. 加上公共菜单

补充规则：
- 子菜单缺父级目录时自动补齐目录链
- 同一菜单被多个包绑定时按并集生效

## 8. 权限最终规则
权限最终计算逻辑固定为：

1. 先看当前上下文已生效功能包
2. 展开出权限候选范围
3. 叠加角色/用户减法裁剪
4. 得到最终 `permission_key` 集合
5. 后端中间件直接按最终集合校验

补充规则：
- 同一权限被多个包绑定时按并集生效
- 某个包被移除，只要另一个包仍提供同一权限，则该权限继续生效

## 9. 自动注册边界
当前正式边界是：

- `permission_key` 自动注册
- 菜单在全量菜单页维护
- 功能包手工创建
- 功能包手工绑定菜单和权限
- 组合包手工绑定基础包

结论：
- 自动化负责基础定义
- 人工配置负责业务组合

## 10. 初始化规则
系统需要内置初始化包，避免系统管理员第一次登录后没有权限配置入口。

正式方向：
- 一组内置基础包
- 一个或多个内置组合包
- 自动赋给系统管理员角色
- 可编辑内容，不可删除

## 11. 性能策略
为了满足“高效”，运行时不应每次临时全链路现算。

当前正式方向：
- 先做数据库快照表
- 后续再考虑加内存缓存

快照重点：
- 角色生效包快照
- 用户最终菜单快照
- 用户最终权限快照

### 11.1 快照刷新机制
快照机制必须作为正式规则维护，不能只停留在实现细节。

核心要求：
- 任何影响功能包展开、菜单显示、最终权限集合的写操作，都必须触发刷新。
- 第一阶段以数据库快照刷新为准，后续再叠加内存缓存。
- `platform` 与 `team` 必须分上下文刷新，不能混用。

至少覆盖以下触发源：
- 功能包绑定关系变更
- 组合包包含关系变更
- 菜单停用或启用变更
- 角色功能包变更
- 用户直绑功能包变更
- 角色减法裁剪变更
- 用户隐藏菜单变更
- 团队开包变更
- 团队边界减法变更
- 团队角色裁剪变更
- 成员角色变更
- 功能包状态变更

### 11.2 当前已接入统一刷新入口的主链
截至 2026-03-23，统一刷新入口已经覆盖：

- 功能包绑定权限、菜单、组合包关系、团队开通关系
- 菜单更新、删除、启停类变更
- 平台角色包、平台角色菜单减法、平台角色权限减法
- 平台用户直绑包、平台用户全局角色分配
- 团队边界权限减法
- 团队成员加入、移除、身份角色变更
- 团队成员角色分配
- 团队角色包、团队角色菜单减法、团队角色权限减法
- 功能权限删除后的关联主体刷新

当前仍是兼容入口而非主链的点：

- `platform` 下的 `user_action_permissions` 仍保留为“用户权限例外”入口。
- 平台用户主链已经转向“全局角色包并集 + 用户直绑包 + 用户隐藏菜单”；旧入口只在回退路径继续参与。

正式要求：
- 不允许依赖“用户重新登录后自然修复”作为主刷新机制。
- 新增一类授权关系时，必须同步补快照刷新触发点。
- 写链统一进入刷新入口，避免各模块各自补丁式刷新。
- 刷新必须在写事务成功后触发，失败事务不得污染快照。

## 12. 当前这份设计对系统意味着什么
这套设计落地后，权限系统会从“权限管理页到处勾”变成“先开包，再裁剪”。

也就是：
- 平台或团队先拿到模块
- 角色再决定模块里谁能看见什么、谁能做什么
- 后端始终只认最终展开后的 `permission_key`

这会让系统更符合：
- 完全
- 高效
- 配置简单
- 注册自动化
- 不冗余

## 13. 当前最值得继续推进的实现方向

### 第一优先级
- 按本设计补齐平台角色、全局用户、团队边界、团队角色的完整功能包链路

### 第二优先级
- 把团队成员个人例外逻辑彻底退出主链
- 把旧的零散权限直配入口降级或移除

### 第三优先级
- 正式引入基础包与组合包
- 建立稳定快照刷新机制

## 14. 结论
当前权限系统设计已经明确：

- 功能包是唯一业务开通入口
- 菜单和权限都围绕功能包展开
- 角色和用户只负责在功能包范围内做减法裁剪
- 平台与团队是唯一上下文
- `permission_key` 是底层运行时判定键

后续代码和数据库调整，都应以这套设计为唯一基线。
