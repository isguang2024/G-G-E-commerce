## 二、近期可做 (Sprint Backlog)

### 2.1 Step-up Authentication (升级认证)

**场景**: 用户已登录，但执行敏感操作（如修改密码、支付）时需要额外验证。

**方案**:
- 新增 `step_up` 认证等级概念，在 capabilities.auth 中声明
- 敏感 API 端点通过中间件检查 `auth_time` + `max_age` 判定是否需要重认证
- 前端弹出确认对话框或跳转到认证中心（带 `prompt=login&max_age=0`）
- 成功后刷新 token，继续原操作

**涉及文件**:
- `backend/internal/api/middleware/` — 新增 step-up 中间件
- `frontend/src/domains/auth/` — 新增 step-up 确认组件

### 2.2 Component-driven 模板实际接入

**场景**: 当前 config-driven 模板可通过 JSON 配置覆盖 80% 场景；component-driven 模板（独立 Vue 文件）的加载机制已预留但未实际使用。

**方案**:
- 编写 2-3 个示例 component-driven 模板放在 `views/auth/templates/`
- 通过 `defineAsyncComponent` + 动态 import 按 template_key 加载
- 管理界面支持上传/编辑 component-driven 模板的元数据

### 2.3 注册页主题共享

**场景**: 注册页和登录页应共享同一套视觉主题（品牌色、Logo、背景图）。

**方案**:
- 在 `login_page_templates` 表中复用 `config` JSON 的 `theme` 字段
- 注册页通过 `useAuthPageTemplate` 读取同一 loginPageKey 的主题配置
- `GetRegisterContext` 返回值追加 `theme` 字段

### 2.4 401 响应携带 login_page_key

**场景**: 业务 API 返回 401 时，前端刷新 token 失败后需跳转登录页。当前跳转不带 login_page_key，导致无法匹配自定义模板。

**方案**:
- `auth-session.ts` 中 `triggerUnauthorizedLogout` 增加对当前 APP 的 loginPageKey 解析
- 跳转认证中心时携带 `login_page_key` 参数

## 三、中期规划 (1-2 Sprints)

### 3.1 Social Login (社交登录/第三方登录)

**目标**: 支持微信、GitHub、Google 等第三方 OAuth 登录。

**架构设计**:
```
用户 → 认证中心登录页 → 选择第三方 → OAuth 授权 → 回调到认证中心
→ 绑定/创建用户 → 签发 callback code → 回跳业务 APP
```

**关键决策**:
- 第三方 OAuth 回调统一由认证中心处理，不由各业务 APP 分别对接
- `login_page_templates` 的 `config.features` 声明启用哪些社交登录按钮
- 新增 `social_accounts` 表存储第三方账号绑定关系
- 新增 `auth_providers` 表配置各 OAuth provider 的 client_id/secret

**涉及文件**:
- `backend/api/openapi/domains/auth/` — 新增 `/auth/social/{provider}` 端点
- `backend/internal/modules/system/auth/` — 新增 social auth service
- `frontend/src/domains/auth/` — 社交登录按钮组件

### 3.2 登录模板版本管理

**目标**: 支持模板修改历史、回滚、A/B 测试。

**方案**:
- `login_page_templates` 新增 `version` 字段
- 新增 `login_page_template_versions` 表存储历史版本
- 发布新版本时旧版本归档，可一键回滚
- A/B 测试通过 `config.ab_test` 字段按比例分流

### 3.3 Captcha 体系完善

**目标**: 当前注册流程支持 captcha，登录流程尚未接入。

**方案**:
- 登录页模板 `config.features.captcha` 控制是否启用
- 支持多种 captcha provider (reCAPTCHA, hCaptcha, 极验)
- 登录失败 N 次后自动启用 captcha (渐进式)

## 四、长期方向 (季度级)

### 4.1 跨域 SSO (Cross-Domain SSO)

**场景**: 业务 APP 分布在不同域名下（如 `admin.example.com` / `shop.example.com`），需要跨域共享登录态。

**方案选型**:
- **短期**: 通过认证中心 redirect 链实现（当前架构已支持）
- **中期**: 引入 iframe + postMessage 的 session check 机制
- **长期**: 考虑引入标准 OIDC Provider，作为独立的 IdP 服务

**风险**: 第三方 cookie 限制趋严，需要关注浏览器策略变化。

### 4.2 Multi-Factor Authentication (MFA)

**目标**: 支持 TOTP、SMS、Email 等多因素认证。

**架构设计**:
```
login → 密码验证通过 → 检查 MFA 策略 → 需要 MFA
→ 返回 mfa_required + mfa_token → 前端展示 MFA 输入
→ 验证 MFA code → 签发完整 token
```

**关键决策**:
- MFA 策略在 APP 级配置 (`capabilities.auth.mfa_policy`: none/optional/required)
- 新增 `user_mfa_methods` 表存储用户绑定的 MFA 方式
- Login 响应新增 `mfa_required` 状态，前端路由到 MFA 验证页
- `auth_time` 已在 JWT 中，MFA 验证后更新 auth_time

### 4.3 Session 集中管理

**目标**: 管理员可查看、强制下线用户会话；用户可管理自己的活跃设备。

**方案**:
- 新增 `active_sessions` 表 (user_id, device_info, ip, created_at, last_active_at)
- JWT 签发时写入 session 记录，refresh 时更新 last_active_at
- 管理后台提供会话列表和强制吊销功能
- 用户个人中心展示"登录设备管理"

### 4.4 租户级认证策略隔离

**目标**: 不同租户可配置独立的认证策略（密码策略、MFA 要求、IP 白名单等）。

**方案**:
- 新增 `tenant_auth_policies` 表
- 认证流程中根据 tenant_id 加载对应策略
- 管理后台提供租户级认证配置界面
- 兼容当前 `default` 单租户模式

## 六、架构约束

1. **OpenAPI 为唯一真相源**: 所有新端点必须先声明 OpenAPI spec，再生成代码
2. **认证中心统一收口**: 第三方登录、MFA 验证等新能力均由认证中心处理
3. **向后兼容**: 新增字段均带默认值，不影响已有 APP 行为
4. **租户感知**: 所有查询必须带 tenant_id 过滤
5. **capabilities JSONB 扩展**: 新增认证相关配置优先放在 `capabilities.auth` 下
