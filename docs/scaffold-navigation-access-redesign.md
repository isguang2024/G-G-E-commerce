## 2. 全链路体检

### 2.1 工作台链路

当前现状：
- 工作台由 `Dashboard -> Console` 菜单入口直接承载。
- 个人中心、消息中心属于非侧栏常驻访问页，其中个人中心是受管页面，消息中心历史上同时存在菜单入口与页面记录兼容残留。

问题：
- `workspace.inbox` 历史上既当菜单入口又当页面记录，容易形成重复注册与扫描误判。
- 工作台相关页本质上是“首页入口 + 少量全局内页”，不该继续拆成菜单/页面双写。

结论：
- 工作台类页面以后统一按“入口菜单 + 少量全局受管页”建模。

### 2.2 消息链路

当前现状：
- 平台消息发送、团队消息发送已有入口菜单。
- 模板、发送人、接收组、记录属于受管页面，挂接到对应消息入口下。

问题：
- 历史上 `system.message.manage`、`team.message.manage` 同时作为入口菜单和页面记录存在。
- 页面管理和菜单管理对“消息主页面”的描述曾长期重复。

结论：
- 消息主入口回归菜单 entry。
- 模板/记录/配置页继续留在页面中心，作为挂接受管页面维护。

### 2.3 身份权限链路

当前现状：
- 权限键、功能包、角色、用户、团队边界已收口到快照读链。
- 前端路由守卫历史上还会重复读取 `requiredAction/requiredActions` 做显隐拦截。

问题：
- 后端已经能给出菜单可见性和页面可访问性，前端再做一轮“路径是否有权限”判断，链路偏重。
- 登录首屏曾存在“先菜单、再页面、再前端过滤”的重复计算。

结论：
- 权限底座不重写。
- 这轮只把“导航 / 页面 / 前端守卫”的重复判断收敛到后端一次编译。

### 2.4 导航与页面治理链路

当前现状：
- 菜单和页面均已有管理页。
- 菜单类型已开始向 `directory / entry / external` 收口。
- 页面运行时已开始过滤菜单直达重复页。

问题：
- 页面管理历史上承担了部分“入口页登记”职责，导致复用页面要按空间重复创建。
- 未注册页面扫描虽然已排除大部分菜单组件，但管理端心智还不够明确。
- 菜单管理中“受管页面关系”仍是兼容展示，不应再是主操作。

结论：
- 菜单管理只维护导航树与入口路由。
- 页面管理只维护非菜单直达页。

### 2.5 接口注册链路

当前现状：
- API 注册、权限键绑定、接口状态缓存已经稳定。
- 本轮新增 `GET /api/v1/runtime/navigation`，作为前端首屏唯一导航 manifest。

问题：
- 旧接口 `/menus/tree`、`/pages/runtime` 仍存在兼容读取路径。
- 管理端部分页面仍在前端自行拼默认首页候选，而不是直接消费统一 manifest。

结论：
- 旧接口保留一版兼容，但内部必须逐步统一到同一编译结果。
- 后台管理页涉及入口候选时优先复用 manifest，而不是再发两套请求拼装。

### 2.6 团队治理链路

当前现状：
- 团队管理入口已经改回菜单 entry，`/team/team` 不再要求必须有页面记录。
- 团队角色权限、团队消息、团队成员页已形成“团队首页入口 + 受管子页”的雏形。

问题：
- 历史 `team.team` 是最典型的菜单/页面同路径冲突样例。
- 多空间语义下，团队侧若继续复制页面定义，会让复用成本飙升。

结论：
- 团队治理链路作为首批标准样例，必须坚持“入口回菜单、子页回页面中心”的分层。

## 3. 根因归纳

### 3.1 页面空间化过重

旧模型把 `ui_pages.space_key` 当成页面主语义，导致：
- 同一组件要在多个空间重复建页面定义
- 页面管理和菜单空间管理强耦合
- 复用页面、流程页、详情页维护成本高

### 3.2 双轨路由源混用

旧链路里：
- 菜单可以直接指向组件
- 页面也可以指向同一路径和组件
- 前端还会把两者再合并注册

结果是：
- 同路径冲突
- 未注册扫描误判
- 面包屑和高亮链容易分叉

### 3.3 重复权限判断

旧流程中既有：
- 快照 / 边界 / 功能包编译菜单
- 页面运行时过滤
- 前端守卫再按路径和动作做二次裁剪

结果是：
- 接口已经返回可见，前端仍可能过滤掉
- 首屏请求偏多，行为排查困难

### 3.4 空间 / 菜单 / 页面边界混乱

旧模型把“空间里有什么”“菜单树长什么样”“页面是否可访问”混在一起，表现为：
- 空间初始化复制页面定义
- 菜单管理里持续出现“主页面”概念
- 页面管理里持续强调按空间建页

## 4. 目标架构

### 4.1 显式双模型

#### 菜单入口路由

菜单只保留三类：
- `directory`：仅用于导航分组和层级
- `entry`：菜单叶子自己就是页面入口，直接持有 `path/component/accessMode`
- `external`：仅用于外链跳转，直接持有 `link`

核心规则：
- 入口菜单不再要求额外创建 `ui_pages`
- 菜单管理不再把“挂接主页面”当主操作

#### 受管页面

页面中心只保留非菜单直达页：
- `inner`
- `global`
- `group`
- `display_group`

适用对象：
- 详情页
- 流程页
- 独立全局页
- 菜单入口下的配置页 / 模板页 / 记录页
- 共享内页

### 4.2 空间边界

空间只负责：
- 菜单树归属
- 默认首页
- Host 绑定
- 极少数独立页暴露

页面默认全局定义：
- 有菜单父级：从菜单继承空间
- 有页面父级：从父页面继承空间
- 无菜单父级且无页面父级：默认全局共享
- 少数必须限制空间的独立页：通过 `page_space_bindings` 做显式暴露

### 4.3 统一访问编译层

后端统一引入请求级编译上下文：
- `ResolvedAccessContext`
- `CompiledAccessContext`
- `AccessGraphCompiler`（本次代码层目前以统一编译语义落在 page/navigation 服务中）

统一输出：
- 当前空间解析结果
- 已裁剪菜单树
- 入口菜单路由
- 受管页面路由
- 动作权限摘要
- 版本戳

正式入口：
- `GET /api/v1/runtime/navigation`

### 4.4 前端消费方式

前端首屏链路固定为：
1. 拉用户信息
2. 拉 `runtime/navigation`
3. 注册菜单树 + 受管页面路由
4. 进入首屏业务页

前端不再承担：
- 二次菜单显隐裁剪
- 二次页面权限裁剪
- 二次空间归属推导

前端仅保留：
- 轻量归一化
- 路由注册
- 缺失动态路由时的 manifest 重拉
- 未命中时回退/404

## 5. 数据与接口迁移策略

### 5.1 兼容期

保留但降级的内容：
- `/api/v1/menus/tree`
- `/api/v1/pages/runtime`
- `ui_pages.space_key` 兼容读取
- 管理页中的“受管页面关系”只读提示

此阶段目标：
- 新写入按新模型落库
- 旧数据可读，不再继续放大

### 5.2 切换期

必须完成：
- 菜单写接口显式要求 `kind`
- 菜单空间初始化停止复制页面定义
- 清理同路径同组件的菜单/页面重复记录
- 前端守卫切到 `runtime/navigation`
- 管理页文案切换到“入口菜单 / 受管页面 / 独立页暴露”

### 5.3 收口期

后续继续做的事情：
- 旧兼容字段只读，不再对外强调
- 后端旧接口内部统一复用同一编译结果
- 默认首页候选、菜单空间视角等管理页细节全部改为消费 manifest

## 6. 验收矩阵

| 场景 | 验收点 |
| --- | --- |
| 单空间基线 | 平台管理员登录后，只需用户信息 + navigation manifest 即可完成动态路由注册 |
| 默认空间主入口 | 菜单入口页无需页面记录也能访问、刷新、高亮、面包屑正确 |
| 工作台 / 消息 / 系统管理 / 团队管理 | 四条主入口链全部可用 |
| 多空间 | 新空间初始化只复制菜单树与功能包菜单关联，不复制页面定义 |
| 共享页面 | 同一受管页面可在多个空间被不同菜单复用，无需重复建页 |
| Host 绑定 | 未命中 Host 时回退默认空间；显式 `space_key` 与 Host 绑定解析一致 |
| 权限变更 | 刷新一次快照 / 访问编译结果即可同步影响菜单树、受管页面和深链访问 |
| 深链访问 | 入口菜单和受管页面都支持刷新后恢复 |
| 冲突清理 | 菜单/页面同路径冲突告警清零，至少覆盖 `team.team -> /team/team` |
| 后台模块冒烟 | 菜单管理、页面管理、菜单空间、角色、功能包、功能权限、API 管理、团队管理、团队消息完成一轮浏览器检查 |

## 8. Sponge 评估

### 8.1 结论

本次不引入 [Sponge](https://github.com/go-dev-frame/sponge) 作为核心重构底座。

### 8.2 不引入为主框架的原因

1. 本轮改造的核心不是 CRUD 生成，而是导航 / 页面 / 空间 / 权限的主链语义收敛。  
2. 当前系统已经具备自己的权限快照、菜单、页面、API 注册和多上下文治理底座，强行换底会引入更大迁移成本。  
3. Sponge 更适合新建边界清晰、以接口和表单为主的独立模块，不适合直接替换当前核心导航与授权主链。  
4. 当前重构更需要“统一编译 + 统一消费”的运行时协议，而不是重新生成一套目录。  

### 8.3 可以借用的部分

未来在以下场景可以局部借用 Sponge 思路：
- 独立新模块的 CRUD 生成
- DTO / service / repository 目录规范参考
- 文档与代码脚手架模板

前提：
- 仅限边界清晰的新模块
- 不进入当前导航 / 页面 / 权限 / 空间主链

## 9. 最终原则

这次重构按以下优先级取舍：
1. 脚手架优先  
2. 操作简化优先  
3. 内部一致性优先  
4. 多空间可选增强，不干扰单空间默认落地  
5. 菜单入口回菜单，受管页面回页面中心，权限判断回后端一次编译
