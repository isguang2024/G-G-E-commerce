# 功能包优先权限系统设计基线

## 1. 文档定位
本文档是当前权限系统的正式设计基线，用于约束后续数据库、后端、前端和迁移实现。

规则：
- 设计以“功能包优先”为核心。
- 文档描述目标正式模型，不要求当前代码 100% 已全部实现。
- 任何后续实现、重构、迁移，都以本文档为准。

## 2. 设计目标
- 完全：统一覆盖平台、团队、菜单、功能权限、角色、用户。
- 高效：运行时优先命中快照，不依赖全链路现算。
- 配置简单：业务开通以功能包为主，不直接暴露零散权限。
- 自动注册：`permission_key` 自动注册，菜单集中维护。
- 不冗余：明确每一层职责，减少重复配置。

## 3. 上下文模型
系统只保留两个正式上下文：
- `platform`
- `team`

约束：
- 所有非团队、非租户能力统一归入 `platform`。
- 所有团队、租户内能力统一归入 `team`。
- 不再单独保留 `personal` 上下文。
- 用户同时拥有平台能力和团队能力时，默认进入 `platform`。
- 仅在 `team` 上下文下前端才发送 `X-Tenant-ID`。

## 4. 正式对象
### 4.1 原子对象
- 菜单 `menu`
- 功能权限键 `permission_key`

### 4.2 功能包对象
功能包分两类：
- 基础包 `base`
- 组合包 `bundle`

规则：
- 基础包直接绑定菜单和 `permission_key`。
- 组合包只包含基础包。
- 组合包不能再包含组合包。
- 组合包不能直接绑定菜单和 `permission_key`。

### 4.3 授予对象
`platform`：
- 全局角色
- 全局用户

`team`：
- 团队
- 团队角色

明确取消：
- 团队成员个人不直接绑定团队功能包。

补充规则：
- 团队与团队角色都允许绑定 `team` 与 `platform,team` 功能包。
- 纯 `platform` 功能包不能进入团队授予链路。

## 5. 功能包职责
功能包是唯一业务开通单元。

功能包负责：
- 决定是否拥有某个业务模块。
- 决定菜单候选范围。
- 决定权限候选范围。
- 决定角色、用户后续可以在什么范围内做减法裁剪。

功能包不负责：
- 直接作为最终后端鉴权键。
- 维护 API 注册。
- 维护按钮动作。
- 创建菜单定义。

## 6. 权限键职责
`permission_key` 保留，但职责收敛为：
- 自动注册的原子能力定义。
- 功能包绑定的原子权限项。
- 角色关闭能力时的最小粒度。
- 后端运行时最终鉴权键。

格式规则：
- 正式 `permission_key` 统一使用点式命名，例如 `system.user.manage`。
- 历史冒号格式仅作兼容输入，进入运行时、接口输出与前端判断前都应归一化为点式 key。

正式结论：
- 业务开通不再直接面向零散 `permission_key`。
- 运行时仍然按 `permission_key` 判定。

## 7. 菜单职责
菜单由全量菜单页统一维护。

功能包只绑定已有菜单，不创建菜单。

规则：
- 功能包可绑定目录、页面菜单、外链菜单。
- 按钮不进入功能包，按钮仍由 `permission_key` 控制。
- 同一个菜单允许被多个功能包复用。
- 菜单停用优先级高于功能包绑定。
- 若只绑定了子菜单，系统自动补齐父级目录链。

### 7.1 公共菜单
保留正式概念：
- 公共菜单

规则：
- 不依赖任何功能包。
- 仍受菜单启用/停用控制。
- 适用于首页、公共入口、兜底入口等。

## 8. 功能包与上下文
功能包定义支持：
- `platform`
- `team`
- `platform,team`

但授予和生效必须严格按上下文隔离：
- 平台侧授予只影响平台空间。
- 团队侧授予只影响团队空间。
- 同一个包可支持双上下文，但不会跨上下文穿透。

## 9. 继承与默认规则
### 9.1 角色拿到功能包后的默认行为
- 包内菜单默认全开。
- 包内权限默认全开。
- 后续只做减法裁剪。

### 9.2 多角色合并规则
用户拥有多个角色时，最终结果全部按并集计算：
- 功能包并集
- 菜单并集
- 权限并集

### 9.3 平台侧规则
平台最终包：
- 全局角色功能包并集
- 加上用户直绑功能包

平台允许：
- 角色级菜单减法
- 角色级权限减法
- 用户级菜单减法

平台不再保留正式“用户级权限加减法”主链。
`user_action_permissions` 仅作为兼容审计输出存在。

### 9.4 团队侧规则
团队开通功能包是团队上限。

团队层只做减法边界：
- 团队屏蔽菜单
- 团队屏蔽权限

团队角色规则：
- 基础团队角色为固定内置角色，不可删除、不可改编码。
- 基础团队角色默认继承团队全部可用包。
- 团队自定义角色默认继承团队全部可用包，但允许裁剪成子集。
- 团队角色不能绑定团队未开通的功能包。
- 基础团队角色的功能包页应只读展示当前团队全部可用功能包及其继承结果，不能只显示已继承子集。

团队成员个人层：
- 已退出主链。
- 仅保留兼容审计能力。

## 10. 最终菜单计算
### 10.1 platform
1. 计算平台最终包。
2. 展开菜单候选范围。
3. 过滤停用菜单。
4. 扣减角色隐藏菜单。
5. 扣减用户隐藏菜单。
6. 合并公共菜单。

### 10.2 team
1. 计算团队已开通功能包。
2. 扣减团队屏蔽菜单。
3. 角色从剩余可用包中取子集。
4. 多角色结果并集。
5. 过滤停用菜单。
6. 合并公共菜单。

补充：
- 团队整队快照只表示团队边界，不直接代表成员最终菜单。
- 成员最终菜单需要在团队快照基础上再结合成员角色子集求交。

## 11. 最终权限计算
### 11.1 platform
1. 计算平台最终包。
2. 展开权限候选范围。
3. 扣减角色关闭权限。
4. 得到最终 `permission_key` 集合。
5. 后端中间件按最终集合判定。
6. 平台上下文鉴权应直接读取平台用户快照，不再按是否存在包配置回退旧正向权限表。
7. 平台角色边界读取与写入应统一基于角色快照边界，不再回退 `role_menus` 与 `role_action_permissions` 正向直配链。

### 11.2 team
1. 计算团队已开通功能包。
2. 扣减团队屏蔽权限。
3. 角色从剩余可用包中取子集。
4. 多角色结果并集。
5. 得到最终 `permission_key` 集合。
6. 后端中间件按最终集合判定。
7. 团队上下文鉴权与权限快照输出应直接使用团队角色快照结果，不再回退团队成员个人例外或旧角色正向权限表。

## 12. 自动注册与人工配置边界
自动注册：
- `permission_key`

人工配置：
- 全量菜单定义
- 功能包创建
- 功能包绑定菜单
- 功能包绑定权限
- 组合包绑定基础包
- 团队开通功能包
- 角色绑定功能包
- 菜单减法裁剪
- 权限减法裁剪

原则：
- 自动化负责基础定义。
- 后台配置负责业务组合与授权分发。
- 自动注册写入 `permission_actions` 时必须显式写入正确的 `context_type`，不能依赖模型默认值 `team`；映射缺失时才允许按点式 `permission_key` 前缀规则兜底推导。

## 13. 初始化规则
系统需要内置初始化包，避免系统管理员首次登录后没有入口。

初始化结构：
- 多个内置基础包
- 一个或多个内置组合包

规则：
- 初始化包自动创建。
- 默认赋给系统管理员角色。
- 内置包可编辑内容，但不可删除。
- 普通包可删除、可停用。

当前要求：
- 默认内置包必须同时绑定权限和菜单。
- 默认角色必须自动绑定对应内置包。

## 14. 功能包状态
功能包必须支持：
- 启用
- 停用

规则：
- 包停用后，即使仍被角色、团队、用户绑定，也整体不生效。

## 15. 快照与性能策略
第一阶段正式方案：
- 以数据库快照表为主
- 不依赖全链路现算
- 后续再按需要叠加内存缓存
- 快照缺失时自动刷新并写回快照表，避免将临时现算结果长期留在主读链。

当前正式快照对象：
- `platform_user_access_snapshots`

## 2026-03-24 上下文与权限键补充约束
- 权限自动注册写入 `permission_keys` 时，必须显式写入 `context_type`；平台系统权限不得依赖模型默认值 `team`。
- 当 `context_type` 缺失时，只允许按点式 `permission_key` 与模块编码做兜底推导：
  - `system.*`
  - `tenant.*`
  - `platform.*`
  - 模块 `role / user / menu / menu_backup / permission_key / api_endpoint / feature_package`
  统一归 `platform`
- 权限列表、角色边界、团队边界、功能包来源等正式接口返回中，`permission_key` 必须优先输出点式格式；历史冒号格式只能作为兼容输入，不再作为展示主值。
- `platform_role_access_snapshots`
- `team_access_snapshots`

当前语义：
- 平台用户快照保存平台用户最终包、候选菜单、最终菜单、候选权限、最终权限及来源映射。
- 平台角色快照保存平台角色最终包、候选菜单、最终菜单、候选权限、最终权限及来源映射。
- 团队快照保存团队边界级别的包展开结果、屏蔽结果和最终可用范围。

## 16. 快照刷新机制
快照系统的核心风险不是规则错误，而是刷新不全。

正式规则：
- 所有会影响包展开、菜单显示、最终权限集合的写操作，都必须触发刷新。
- 刷新必须发生在写事务成功之后。
- 不允许依赖“下次登录自动修复”作为主策略。
- 平台与团队上下文必须分别刷新。

### 16.1 平台侧触发源
- 功能包绑定权限关系变更
- 功能包绑定菜单关系变更
- 组合包包含关系变更
- 菜单启用/停用变更
- 平台角色绑定功能包变更
- 平台用户直绑功能包变更
- 平台角色隐藏菜单变更
- 平台角色关闭权限变更
- 平台用户隐藏菜单变更
- 用户全局角色变更
- 功能包状态变更

### 16.2 团队侧触发源
- 功能包绑定权限关系变更
- 功能包绑定菜单关系变更
- 组合包包含关系变更
- 菜单启用/停用变更
- 团队开通功能包变更
- 团队屏蔽菜单变更
- 团队屏蔽权限变更
- 团队角色绑定功能包变更
- 团队角色菜单裁剪变更
- 团队角色权限裁剪变更
- 成员角色变更
- 团队成员加入/移除
- 功能包状态变更

### 16.3 刷新粒度
建议至少支持：
- 刷新单个平台用户快照
- 刷新单个平台角色及其关联用户快照
- 刷新单个团队边界快照
- 刷新单个团队角色及其关联成员结果
- 按功能包反查受影响主体并批量刷新

## 18. 新模块接入顺序
新增模块时固定按以下顺序：
1. 定义 `permission_key`
2. 在全量菜单页定义菜单
3. 创建基础包
4. 绑定菜单和权限到基础包
5. 如有需要，创建组合包并组合基础包
6. 平台或团队开通功能包
7. 角色在功能包范围内做减法裁剪
8. 最后接入 API 和前端显示

### 18.1 新增模块前仍需补齐的基础能力
- 如果希望后续模块直接接入组合包运营，必须先完成组合包关系管理 API 和前端页面。
- 如果希望权限系统进入稳定扩展期，建议先补一份开发清单：
  - 新模块属于平台、团队还是双上下文
  - 是否需要新建基础包
  - 是否需要纳入某个组合包
  - 需要触发哪些快照刷新
  - 需要配置哪些默认角色/默认团队开通策略

## 19. 结论
当前正式模型已经固定为：
- 功能包优先
- 菜单与权限都围绕功能包展开
- 平台与团队是唯一上下文
- 团队开包是上限，角色做减法裁剪
- `permission_key` 是底层运行时鉴权键
- 快照是主读链方向

## 补充说明：功能包上下文与旧数据清理
- 功能包上下文仍然需要区分 `platform` 和 `team`，因为授予对象、生效范围、继承链和运行时计算都不同。
- 但功能包不必强制一包一上下文，正式支持 `platform`、`team`、`platform,team` 三种取值。
- `platform,team` 表示同一个功能包模板可同时用于平台和团队，但授予关系与生效结果必须按上下文独立计算，不能互相穿透。
- 历史遗留的 `resource:action` 权限键已开始统一迁移为点式 key；未知旧键也不再继续生成冒号格式。
- 历史遗留的旧团队菜单树已纳入迁移清理范围，避免与当前 `TeamRoot -> TeamManagement / TeamMembers` 结构重复。

## 2026-03-24 初始化部署补充基线
### 1. 正式授权主链
- 初始化后系统正式授权主链固定为：`api_endpoints -> api_endpoint_permission_bindings -> permission_keys -> permission_packages -> *_feature_packages`。
- 其中“功能键”是管理员可维护的最小授权单元，一个 API 可以归属多个功能键。
- 功能包是功能键集合；角色、用户、团队只绑定功能包，不再新增“角色直绑 API”正式链路。

### 2. 现阶段表结构收口方向
- 直接保留：
  - `api_endpoints`
  - `api_endpoint_categories`
  - `api_endpoint_permission_bindings`
  - `feature_packages`
  - `feature_package_bundles`
  - `role_feature_packages`
  - `user_feature_packages`
  - `team_feature_packages`
- 继续保留为减法/边界辅助，不作为主授权来源：
  - `role_hidden_menus`
  - `user_hidden_menus`
  - `team_blocked_menus`
  - `role_disabled_actions`
  - `team_blocked_actions`
- 后续计划迁移替换：
  - `permission_actions` -> `permission_keys`
  - `feature_package_actions` -> `feature_package_keys`
- `user_action_permissions` 仅保留兼容/审计语义，不再参与新主链设计。

### 3. 初始化主数据规则
- 初始化脚本必须预置基础功能键、管理员基础功能包、管理员角色与管理员功能包绑定关系。
- 默认管理员首次登录后应直接具备后台正常访问和管理 API 注册、功能键、功能包的能力。
- 后续新增业务模块时，项目迁移只需导入内置 Seeds，再自动扫描当前项目路由完成 API 注册。

### 4. ID 固定规则
- 默认主数据必须使用固定 UUID，不依赖数据库自增或随机生成。
- 固定 UUID 的适用对象至少包括：
  - 内置功能键
  - 内置功能包
  - 内置 API 分类
  - 必要的默认 API endpoint code
- 固定 UUID/Code 的目标是保证新环境初始化、历史环境补种、跨环境导入时联表关系稳定。

### 5. API 元数据正式策略
- API 元数据以“代码注册时内联声明”为正式方案，不采用外部配置文件全量维护。
- 原因：
  - 路由与 API 元数据天然强耦合。
  - 新增接口时最不容易漏。
  - 更容易保证“代码真实路由”和“数据库 API 注册表”一致。
- 但默认功能键、默认功能包、默认角色绑定等初始化主数据仍由 Seeds 独立维护。
- 自动注册策略再补充：
  - 只有带元数据的 API 参与自动注册。
  - 没有元数据的普通 API 默认不进入 API 单元。
  - 管理员可以在 API 管理页手动创建 API 单元，并绑定到真实路由。
  - API 管理页必须支持“搜索未注册 API”，从运行时路由中筛出还未进入 API 单元的接口。

### 6. API 元数据 Builder 方向
- 在 `internal/pkg/apiregistry` 内新增链式 Builder，作为后续统一元数据声明方式。
- 目标体验示例：
  - `Meta("创建用户").绑定分组("user").绑定来源("sync").绑定功能键("system.user.manage").Build()`
- Builder 负责：
  - 绑定分组/分类
  - 绑定来源
  - 绑定功能键
  - 推导稳定 endpoint code
  - 统一做默认值与去重
- `Registrar` 继续负责 HTTP 路由注册，Builder 只负责生成 `RouteMeta`，以兼容旧调用并逐步替换重复写法。
- `RouteMeta` 正式字段收口为 `code/module/summary/feature_kind/category_code/context_scope/source/permission_keys`。
- 历史 `resource_code / action_code` 语义不再进入 API 元数据 Builder；API 与功能权限、功能包的关联统一通过 `permission_key` 建立。

### 6.1 功能权限主数据补充
- 功能权限主数据已进一步从历史 action 兼容层收口为“权限键 + 分组”模型。
- `permission_actions` 当前正式使用字段应以：
  - `permission_key`
  - `module_code`
  - `module_group_id`
  - `feature_group_id`
  - `context_type`
  - `is_builtin`
  为准。
- 历史 `source / resource_code / action_code` 已退出正式模型，不应再出现在迁移、接口返回和前端配置主链中。

### 7. 部署初始化 Builder 方向
- 额外提供独立的部署初始化 Builder，用于按链式方式组织默认数据初始化，而不是把部署逻辑写进路由注册点。
- 正式职责包括：
  - 装载默认 API 分类
  - 装载默认功能键
  - 装载默认功能包
  - 装载功能包菜单绑定
  - 装载功能包功能键绑定
  - 装载组合包包含关系
  - 装载默认角色绑定
  - 扫描并同步带元数据的 API
  - 统计未注册路由，供后台手动补录
- 设计要求：
  - 以简单、稳定为第一优先级
  - 要有可观测性，能输出初始化摘要
  - 要允许后期管理员主动补录 API 单元，而不是强依赖开发期元数据完整率
- 默认主数据的正式单一来源应为 `internal/pkg/permissionseed`，迁移入口只负责消费这些 Seeds，不应继续维护一套平行的硬编码默认清单。

### 7.1 初始化执行能力补充
- `permissionseed` 现在应同时提供两层能力：
  - Seeds：描述默认主数据本身
  - Ensure：将默认主数据按固定 ID 写入数据库
- 当前已收口进 `permissionseed` 的执行项包括：
  - 默认 API 分类
  - 默认功能权限分组
  - 默认功能键
  - 默认功能包
  - 默认组合包关系
  - 默认角色功能包绑定
- `cmd/migrate/main.go` 后续应继续只保留“顺序编排 + 日志 + 刷新”，不再承载成片的种子写库细节。

### 8. API 管理页补录交互约束
- API 管理页必须提供“未注册 API”入口，用于查询运行时已存在但尚未进入 `api_endpoints` 的路由。
- 未注册路由查询至少支持：`method/path/module/keyword/only_no_meta` 五组筛选参数。
- 管理员选择某条未注册路由后，系统应允许“一键带入新增 API 表单”，而不是要求手动重复填写方法、路径、模块等基础信息。
- 若路由自身带元数据，则新增表单默认继承：
  - `summary`
  - `module`
  - `category_code`
  - `context_scope`
  - `source`
  - `feature_kind`
  - `permission_keys`
- 若路由不带元数据，则默认按手工补录处理，来源应落入 `manual`。
- API 管理首页应有轻量状态概览，用于直接展示自动注册、手工补录、初始种子、未注册路由四类数量，作为部署后排查和运营入口的第一层观察面板。
- API 主列表还应提供按注册方式的快速筛选能力，让管理员可以直接切换查看 `sync/manual/seed` 三类 API，而不是只在详情列中被动识别。
- API 管理页应补齐统一查询条，至少支持 `method/path/category/contextScope/status` 五类条件，与模块切片、来源切片共同组成标准筛选入口。

### 8.1 内置功能包迁移覆盖规则补充
- 内置功能包允许管理员在运营期继续调整内容，但不允许被删除。
- 因此迁移阶段对内置功能包的默认数据策略必须是：
  - 缺失默认关系时补齐
  - 已存在关系时保留
  - 不再通过“先删后重建”的方式强制覆盖整包动作、菜单或组合包关系
- 当前默认 Seeds 导入的正式规则：
  - `feature_package_actions` 只补缺失默认动作
  - `feature_package_menus` 只补缺失默认菜单
  - `feature_package_bundles` 只补缺失默认组合关系
- 默认角色与功能包绑定也应保持固定 ID 能力，用于新环境初始化一致性；首次创建时写入固定 UUID，已存在时至少保证同一 `(role_id, package_id)` 关系可持续复用。
- 功能包上下文约束继续保持严格校验：
  - 平台角色只允许绑定 `platform` 或 `common` 上下文功能包
  - 团队角色与团队边界只允许绑定 `team` 或 `common` 上下文功能包
  - `common` 功能包在功能范围选择时应同时允许看到 `common` 功能权限本身

### 8.2 功能包功能键关联表命名补充
- 功能包与功能键的正式关联表名已收口为 `feature_package_keys`。
- 历史表名 `feature_package_actions` 仅保留迁移兼容意义，不再作为正式设计文档中的表名。
- 功能权限主表的正式名称同步收口为 `permission_keys`，因此功能包关系链正式表述应为：
  - `permission_keys`
  - `feature_package_keys`
  - `feature_packages`
  - `role_feature_packages / user_feature_packages / team_feature_packages`
- 当前代码层的正式命名也已同步收口：
  - `PermissionKey`
  - `FeaturePackageKey`
  - `PermissionKeyRepository`
  - `FeaturePackageKeyRepository`
- 当前仅保留外部兼容命名：
  - 路径层 `/actions`
  - JSON 字段 `action_ids / action_id`
  这些兼容项仅用于维持前端与既有调用稳定，不代表内部设计仍使用旧 action 语义。

### 8.3 默认权限分组固定化补充
- 默认功能键现在必须绑定固定的模块分组与功能分组，不能再在部署时通过现有权限键列表反推模块分组名称。
- `permissionseed` 必须显式维护：
  - 默认功能分组清单
  - 默认模块分组清单
  - 每个默认功能键对应的 `module_group_code`
  - 每个默认功能键对应的 `feature_group_code`
- 当前系统内置功能键的默认功能分组统一收口到 `system`，这样在迁移导入时，模块分组和功能分组都能直接一次写库并稳定复用。
- 默认分组 ID 必须使用稳定 UUID，与 API 分类一样预先固定，保证跨环境迁移、导表、联表和初始化快照都不会因重新生成 ID 而漂移。

### 8.4 permission_key 正式编码补充
- 功能权限模块的正式模块编码已经收口为 `permission_key`，不再使用 `permission_action` 作为默认主数据编码。
- 对应的默认主数据必须同步保持一致：
  - API 分类编码使用 `permission_key`
  - 模块分组编码使用 `permission_key`
  - 内置功能键的 `module_code` 使用 `permission_key`
- 老环境迁移时，允许保留少量兼容解析，但数据库最终状态必须只保留 `permission_key` 这一套正式编码。
