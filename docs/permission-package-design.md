# 功能包优先权限系统设计 V1

## 1. 文档定位
这份文档记录当前已经确认的权限系统正式设计基线，用于后续数据库、后端、前端和迁移方案统一落地。

注意：
- 本文档描述的是目标正式模型。
- 不代表当前代码已经 100% 全量实现。
- 后续实现、迁移、收口都应以本文档为准。

## 2. 设计目标
本次权限系统设计固定追求以下目标：

- 完全：覆盖平台、团队、菜单、权限、角色、用户的统一模型。
- 高效：运行时不做全链路临时拼装，优先使用预展开快照。
- 配置简单：业务开通以功能包为主，不再让运营和管理员直接面对零散权限。
- 注册自动化：权限键自动注册，菜单统一配置，功能包手工组合。
- 不冗余：减少重复配置，明确每一层职责。

## 3. 上下文模型
系统只保留两个正式上下文：

- `platform`
- `team`

说明：
- 一切非团队、非租户能力，都归 `platform`。
- 一切团队、租户内能力，都归 `team`。
- 不再单独拆 `personal` 上下文；个人型业务能力归入 `platform`。

## 4. 正式对象

### 4.1 原子对象
- `permission_key`
- `menu`

职责：
- `permission_key` 是最小运行时鉴权单元。
- 菜单是前端页面入口承载体。
- 页面当前不单独建资源表，跟随菜单入口。

### 4.2 功能包对象
功能包分为两类：

- `基础包`
- `组合包`

规则：
- 基础包直接绑定菜单和 `permission_key`。
- 组合包只包含基础包。
- 组合包不能包含组合包。
- 组合包不能直接绑定菜单和 `permission_key`。
- 同一菜单和同一权限允许被多个功能包重复绑定。

### 4.3 主体对象
- 全局角色
- 全局用户
- 团队
- 团队角色

说明：
- 团队成员个人不直接绑定团队功能包。
- 团队成员仅通过团队角色获得团队上下文能力。

## 5. 功能包职责
功能包是唯一业务开通单元。

正式职责：
- 决定某个主体是否拥有某个业务模块。
- 决定某个上下文下有哪些菜单候选范围。
- 决定某个上下文下有哪些权限候选范围。
- 决定角色、用户后续可以从哪些菜单和权限中做减法裁剪。

非职责：
- 不直接参与 API 注册。
- 不直接作为后端最终鉴权键。
- 不负责维护菜单定义本身。
- 不负责维护按钮级别动作。

## 6. 权限键职责
`permission_key` 仍然保留，但职责收敛为：

- 自动注册的原子能力定义。
- 功能包可绑定的原子权限项。
- 角色或用户减法裁剪时的最小权限粒度。
- 后端最终中间件的运行时判断键。

正式结论：
- `permission_key` 不再作为主要人工开通入口。
- 业务开通一律以功能包为主。

## 7. 菜单职责

### 7.1 菜单来源
菜单统一在全量菜单页维护定义。

功能包只做：
- 绑定已有菜单

功能包不做：
- 创建菜单定义
- 编辑菜单定义

### 7.2 菜单粒度
功能包只绑定以下菜单类型：

- 目录
- 页面菜单
- 外链菜单

明确不绑定：
- 按钮

按钮或操作继续走 `permission_key`。

### 7.3 公共菜单
系统保留正式概念：

- `公共菜单`

规则：
- 不依赖任何功能包
- 仍受菜单自身启用/停用控制
- 用于首页、公共入口、系统保底入口等场景

### 7.4 父级目录补齐
如果功能包绑定了子菜单但未绑定父级目录：

- 系统自动补齐父级目录链

目的：
- 保证前端菜单树正常显示
- 减少手工配置成本

### 7.5 菜单启停优先级
如果菜单在全量菜单页被停用：

- 即使功能包绑定了该菜单
- 即使角色或用户保留了该菜单
- 仍然不可显示

## 8. 功能包与上下文
功能包本身可以支持：

- `platform`
- `team`
- `platform,team`

但同一个功能包在不同上下文中的授予与生效必须完全独立。

也就是：
- 平台侧授予，只影响平台空间。
- 团队侧授予，只影响团队空间。
- 互不穿透。

## 9. 授予关系

### 9.1 platform
平台上下文功能包可授予给：

- 全局角色
- 全局用户

### 9.2 team
团队上下文功能包可授予给：

- 团队
- 团队角色

明确不允许：
- 团队成员个人直接绑定团队功能包

## 10. 默认继承规则

### 10.1 角色拿到包后的默认行为
统一规则：

- 包内菜单默认全开
- 包内权限默认全开
- 后续只允许做减法裁剪

该规则统一适用于：
- 平台角色
- 团队角色

### 10.2 多角色并集规则
用户同时拥有多个角色时：

- 功能包按并集
- 菜单按并集
- 权限按并集

即：
- 只要某个角色拥有，就进入最终结果

## 11. platform 上下文正式模型

### 11.1 生效功能包
平台最终包计算：

`平台最终包 = 全局角色包并集 + 用户直绑包`

### 11.2 用户级裁剪
平台用户允许用户级减法裁剪。

正式模型当前先保留一层：
- 用户隐藏菜单

规则：
- 用户直绑包默认全开
- 平台管理员可再做用户级减法裁剪

### 11.3 最终菜单公式
平台最终菜单：

1. 计算平台最终包
2. 展开出菜单候选范围
3. 过滤停用菜单
4. 扣减角色隐藏菜单
5. 扣减用户隐藏菜单
6. 加上公共菜单

### 11.4 最终权限公式
平台最终权限：

1. 计算平台最终包
2. 展开出权限候选范围
3. 扣减角色关闭权限
4. 得到最终 `permission_key` 集合
5. 后端中间件按最终集合校验

## 12. team 上下文正式模型

### 12.1 团队开通包是绝对上限
团队开通功能包后，决定团队上下文最大边界。

团队角色不能超出团队已开通包。

### 12.2 团队边界
团队层正式语义固定为：

- 只做减法边界

也就是：
- 允许平台管理员对团队已开通包做屏蔽
- 不允许团队层额外加出包外能力

团队边界正式模型拆成两层：
- 团队屏蔽菜单
- 团队屏蔽权限

### 12.3 团队角色
基础团队角色：
- 固定内置
- 不可删除
- 不可改角色编码
- 默认继承团队全部可用包
- 允许裁剪包内菜单和权限

团队自定义角色：
- 默认继承团队全部可用包
- 允许裁剪成团队当前可用包的子集
- 不允许绑定团队未开通的包
- 允许裁剪包内菜单和权限

### 12.4 团队成员个人层
团队成员个人层正式取消：

- 不再允许团队成员个人做菜单或权限例外配置
- 团队内能力只由“团队开包 + 团队角色裁剪”决定

### 12.5 最终菜单公式
团队最终菜单：

1. 计算团队已开通包
2. 扣减团队屏蔽菜单对应的包内菜单
3. 角色从团队剩余可用包中取子集
4. 多角色菜单结果取并集
5. 过滤停用菜单
6. 加上公共菜单

### 12.6 最终权限公式
团队最终权限：

1. 计算团队已开通包
2. 扣减团队屏蔽权限对应的包内权限
3. 角色从团队剩余可用包中取子集
4. 多角色权限结果取并集
5. 得到最终 `permission_key` 集合
6. 后端中间件按最终集合校验

## 13. 角色级裁剪模型
角色级减法裁剪统一模型，平台角色与团队角色共用同类设计：

- 角色隐藏菜单
- 角色关闭权限

统一规则：
- 角色只能在自己已生效功能包展开范围内做减法
- 角色不能减到包外对象

## 14. 自动注册与人工配置边界

### 14.1 自动注册
系统自动注册：

- `permission_key`

### 14.2 人工配置
人工配置：

- 全量菜单定义
- 功能包创建
- 功能包绑定菜单
- 功能包绑定权限
- 组合包绑定基础包
- 团队开通功能包
- 角色绑定功能包
- 角色菜单减法裁剪
- 角色权限减法裁剪
- 平台用户菜单减法裁剪

正式原则：
- 自动化负责基础能力定义
- 后台配置负责业务组合和授权分发

## 15. 运行时性能策略
运行时采用预展开快照策略。

第一阶段正式方向：
- 只做数据库快照表

后续如有必要：
- 再叠加内存缓存

建议快照范围：
- 角色生效基础包集合
- 用户在某上下文下的最终菜单集合
- 用户在某上下文下的最终权限集合

原则：
- 写入时刷新
- 读取时直接命中快照

### 15.1 快照刷新机制
快照系统最怕规则正确但刷新不全，因此刷新机制必须单独作为正式规范维护。

正式原则：
- 所有影响“功能包展开结果、菜单可见结果、最终权限结果”的写操作，都必须触发快照刷新。
- 快照刷新必须发生在写事务成功之后；事务失败时不得提前刷新。
- 第一阶段以数据库快照刷新为准，不依赖手工补偿。
- 同一次变更如果会影响多个主体，允许批量刷新，但不能漏刷。
- 刷新链路必须支持按上下文区分：`platform` 与 `team` 分开刷新。

### 15.2 平台上下文刷新触发源
以下变更必须刷新平台侧相关快照：

- 功能包绑定权限变更
- 功能包绑定菜单变更
- 组合包包含关系变更
- 菜单启用/停用变更
- 平台角色绑定功能包变更
- 平台用户直绑功能包变更
- 平台角色隐藏菜单变更
- 平台角色关闭权限变更
- 平台用户隐藏菜单变更
- 用户全局角色变更
- 功能包状态变更

### 15.3 团队上下文刷新触发源
以下变更必须刷新团队侧相关快照：

- 功能包绑定权限变更
- 功能包绑定菜单变更
- 组合包包含关系变更
- 菜单启用/停用变更
- 团队开通功能包变更
- 团队屏蔽菜单变更
- 团队屏蔽权限变更
- 团队角色绑定功能包变更
- 团队角色菜单减法裁剪变更
- 团队角色权限减法裁剪变更
- 团队成员角色变更
- 团队成员加入/移除
- 功能包状态变更

### 15.4 当前已接入统一刷新入口的写链
截至 2026-03-23，以下写链已经接入统一刷新入口：

- 功能包绑定权限、绑定菜单、团队开通关系变更
- 组合包包含关系变更
- 菜单更新、删除、启停类变更
- 平台角色绑定功能包
- 平台角色菜单减法裁剪
- 平台角色权限减法裁剪
- 平台用户绑定功能包
- 平台用户分配全局角色
- 团队边界权限减法变更
- 团队成员加入、移除、身份角色变更
- 团队成员角色分配变更
- 团队角色绑定功能包
- 团队角色菜单减法裁剪
- 团队角色权限减法裁剪
- 功能权限删除后按关联功能包反查刷新

当前仍处于兼容链的点：

- `platform` 下的 `user_action_permissions` 仍保留为“用户权限例外”兼容入口，不是主开通链。
- 平台用户运行时主链已经是“全局角色包并集 + 用户直绑包 + 用户隐藏菜单”；`user_action_permissions` 仅在旧回退路径下继续生效。
- 团队角色隐藏菜单变更
- 团队角色关闭权限变更
- 成员角色变更
- 功能包状态变更

### 15.4 刷新粒度建议
建议刷新时至少支持以下粒度：

- 刷新单个平台用户快照
- 刷新单个平台角色关联用户快照
- 刷新单个团队快照
- 刷新单个团队角色关联成员快照
- 按功能包反查并批量刷新受影响主体

实现要求：
- 新增、更新、删除三类写操作都必须进入统一刷新入口。
- 文档新增触发源时，代码中必须同步补刷新实现。
- 不允许依赖“下次登录自然修正”作为主刷新策略。

## 16. 初始化规则
系统初始化采用：

- 一组内置基础包
- 一个或多个内置组合包

规则：
- 初始化包首次自动生成
- 自动赋给系统管理员角色
- 内置包可编辑内容
- 内置包不可删除
- 普通包可删除

建议初始化至少包含：
- 系统管理基础包
- 菜单管理基础包
- 权限管理基础包
- 角色管理基础包
- 功能包管理基础包
- 一个系统管理组合包

## 17. 功能包状态
功能包需要正式支持：

- 启用
- 停用

规则：
- 包被停用后，即使仍被团队、角色、用户绑定，也整体不生效

## 18. 新增模块接入顺序
新增模块时固定按以下顺序：

1. 定义 `permission_key`
2. 在全量菜单页定义或补齐菜单
3. 创建基础包
4. 绑定菜单和权限到基础包
5. 如有需要，创建组合包并收纳基础包
6. 平台或团队开通功能包
7. 角色在功能包范围内做减法裁剪
8. 最后接 API 与前端展示

## 19. 当前仍需实现的重点
本文档已经定下正式设计，但当前代码实现仍需按此继续收口，优先项是：

- 平台角色全面接入平台功能包
- 平台用户直绑包后的用户级减法裁剪模型
- 团队边界改为正式的“屏蔽菜单 + 屏蔽权限”
- 团队成员个人例外能力彻底退出主链
- 基础包与组合包模型正式落库与界面化
- 运行时快照表与刷新策略统一

## 20. 结论
当前正式设计已经明确为：

- 功能包优先
- 菜单和权限都围绕功能包展开
- 平台与团队是唯一上下文
- 团队是上限，角色是细分，用户只在平台侧允许减法裁剪
- `permission_key` 保留为底层运行时判定键
- 配置以“包”为主，运行时以“最终展开结果”为主
