# 消息系统设计

> 现状基线：2026-03-29。本文用于替换右上角通知组件的 mock 数据方案，定义站内消息系统的正式模型与分阶段落地方式。

## 1. 现状判断

当前右上角消息组件位于：

- `frontend/src/components/core/layouts/art-notification/index.vue`

现状特点：

- 面板分为 `通知 / 消息 / 代办` 三栏
- 数据全部是前端 mock
- 没有后端消息主表、收件箱表、未读计数接口
- 前端已有可复用的 websocket 客户端：`frontend/src/utils/socket/index.ts`

因此当前要做的不是“换几条假数据”，而是补齐正式消息系统。

## 2. 设计目标

### 2.1 产品目标

- 右上角面板变成真实收件箱，而不是演示组件
- 用户能区分“看看就行”的通知和“必须处理”的待办
- 平台、团队、系统动作都能统一进入一套消息中心
- 后续可以接 websocket 推送，但第一阶段不依赖它才能工作

### 2.2 视觉与交互方向

- visual thesis：像后台工作收件箱，不像社交 feed，也不像公告弹窗堆叠
- content plan：顶部计数与筛选、三类消息列表、消息详情动作、进入完整消息中心
- interaction thesis：
  - 右上角面板维持轻量预览，只展示最近几条与未读数
  - 点击消息后优先直接落到业务页或右侧详情抽屉
  - 待办类消息必须给出明确主操作，不做纯文本提醒

## 3. 正式消息分类

消息系统保留当前三大入口，但语义要正式化：

### 3.1 通知 `notice`

用于“以告知为主，不要求即时处理”的内容。

建议包含：

- 平台公告
- 团队公告
- 版本发布
- 安全提醒
- 账单/配额提醒
- 系统维护通知

特点：

- 允许批量已读
- 可以没有发送者头像
- 可以有范围标签，如“平台”“当前团队”

### 3.2 消息 `message`

用于“有人或某个业务对象直接发给你”的内容。

建议包含：

- 团队邀请
- 成员变更通知
- 评论 / @ 提醒
- 审批结果通知
- 功能包开通结果
- API / 页面 / 菜单变更提醒

特点：

- 需要展示发送者
- 可以跳转到具体业务对象
- 可以按“来自谁 / 来自哪个团队”聚合

### 3.3 待办 `todo`

用于“你必须处理”的动作型消息。

建议包含：

- 待审核
- 待确认邀请
- 待补充配置
- 待处理异常
- 待完成权限绑定
- 待执行系统同步

特点：

- 一定要有主操作
- 必须记录处理状态
- 不建议仅靠“已读”解决，应有“完成 / 忽略 / 延后”语义

## 4. 发送者模型

消息发送者不要只做成“用户头像 + 名字”，否则系统消息会很乱。建议正式区分：

### 4.1 发送者类型 `sender_type`

- `system`
- `platform_user`
- `team_user`
- `service`
- `automation`

解释：

- `system`：系统自动生成，如安全提醒、权限快照刷新结果
- `platform_user`：平台管理员或平台运营人员发送
- `team_user`：团队成员、团队管理员发送
- `service`：业务服务发送，如“页面同步器”“API 注册器”
- `automation`：定时任务、工作流、机器人发送

### 4.2 发送者展示规则

- `system`：展示统一系统徽标，不展示个人头像
- `service` / `automation`：展示服务图标与服务名
- `platform_user` / `team_user`：展示头像、昵称、所属团队

### 4.3 发送者字段

消息主表建议包含：

- `sender_type`
- `sender_user_id`
- `sender_name_snapshot`
- `sender_avatar_snapshot`
- `sender_service_key`

说明：

- 发送时保留快照，避免用户改名后历史消息全变
- 如果是服务类发送者，不依赖用户表

## 5. 右上角消息面板设计

## 5.1 保留三栏，不改成更多 tab

右上角区域空间很小，保持：

- `通知`
- `消息`
- `待办`

不要继续往上叠“公告”“安全”“审批”等更多一级 tab。

更细分类放到：

- 标签
- 图标
- 副标题
- 完整消息中心页筛选

### 5.2 面板项结构

每条消息建议统一为：

- 标题
- 副标题
- 时间
- 未读状态
- 来源标签
- 快捷动作（仅待办显示）

其中：

- 通知：显示图标 + 标题 + 范围标签
- 消息：显示发送者头像 + 标题 + 副标题
- 待办：显示状态色 + 标题 + 主按钮

### 5.3 面板行为

- 顶部显示三类未读数
- 支持“全部已读”，但仅对 `notice` 和 `message` 生效
- `todo` 不支持简单全部已读，避免掩盖未处理事项
- 点击“查看全部”进入完整消息中心，不再留空处理器

## 6. 完整消息中心设计

建议新增独立页面：

- `#/workspace/inbox`

该页承担：

- 全量消息列表
- 搜索
- 类型筛选
- 已读 / 未读筛选
- 平台 / 团队范围筛选
- 消息详情
- 待办处理

页面结构建议：

1. 左侧分类
- 全部
- 通知
- 消息
- 待办
- 已归档

2. 中间列表
- 紧凑列表
- 支持未读高亮
- 支持批量操作

3. 右侧详情
- 正文
- 发送者
- 关联对象
- 处理动作

右上角面板只做“预览入口”，完整处理放在消息中心。

## 7. 数据模型设计

第一阶段建议拆成两张主表，而不是一张表硬塞所有状态。

### 7.1 消息主表 `messages`

职责：

- 保存消息内容与发送者信息
- 描述消息类型、范围、优先级、动作入口

建议字段：

- `id`
- `message_type`：`notice | message | todo`
- `biz_type`：如 `team_invite`、`api_sync_result`、`security_alert`
- `scope_type`：`platform | team | user`
- `scope_id`
- `sender_type`
- `sender_user_id`
- `sender_name_snapshot`
- `sender_avatar_snapshot`
- `sender_service_key`
- `title`
- `summary`
- `content`
- `priority`：`low | normal | high | urgent`
- `action_type`：`route | external_link | api | none`
- `action_target`
- `status`：`draft | published | revoked | expired`
- `published_at`
- `expired_at`
- `meta`

### 7.2 收件箱表 `message_deliveries`

职责：

- 每个用户一条收件记录
- 管理未读、已读、归档、待办状态

建议字段：

- `id`
- `message_id`
- `recipient_user_id`
- `recipient_team_id`
- `box_type`：冗余自 `message_type`
- `delivery_status`：`unread | read | archived`
- `todo_status`：`pending | processing | done | ignored`
- `read_at`
- `archived_at`
- `done_at`
- `last_action_at`
- `meta`

### 7.3 为什么不只做一张表

因为一条平台公告可能发给很多人：

- 消息内容是一份
- 每个人的已读状态不同

所以必须拆成：

- 内容
- 投递结果

## 8. 接收范围设计

消息投递范围建议支持：

- 单用户
- 多用户
- 单团队全员
- 指定角色用户
- 平台管理员
- 当前团队管理员

第一阶段不要做“复杂规则引擎”，先支持显式投递：

- 指定用户 ID
- 指定团队 ID
- 指定角色编码

真正写入收件箱时展开成用户级记录。

### 8.1 当前必须支持的三类发送对象

结合当前项目方向，第一阶段必须直接支持：

1. 平台给所有用户发
- 典型场景：平台公告、系统升级、安全提醒
- 投递规则：展开到全部启用用户

2. 平台给团队管理员发
- 典型场景：团队配额提醒、待处理配置、团队边界调整提醒
- 投递规则：按团队成员角色展开到 `team_admin`

3. 团队管理员给指定团队发
- 典型场景：团队公告、团队内待办、成员提醒
- 投递规则：只允许发到自己有权限管理的团队

### 8.2 建议的投递目标字段

建议消息主表增加投递目标描述字段：

- `audience_type`
- `audience_scope`
- `target_tenant_id`
- `target_role_codes`
- `target_user_ids`

推荐枚举：

- `audience_type`
  - `all_users`
  - `tenant_admins`
  - `tenant_users`
  - `specified_users`

- `audience_scope`
  - `platform`
  - `tenant`

解释：

- 平台给所有用户发：`all_users + platform`
- 平台给团队管理员发：`tenant_admins + platform`
- 团队管理员给指定团队发：`tenant_users` 或 `tenant_admins` + `tenant`

### 8.3 发送权限边界

第一阶段建议明确：

- 平台管理员
  - 可以给全部用户发
  - 可以给团队管理员发
  - 可以给指定团队发

- 团队管理员
  - 只能给自己可管理的团队发
  - 不能跨团队发
  - 不能发平台全员消息

这部分不要只靠前端限制，后端必须校验发送范围。

## 8.4 模板归属设计

消息模板不应只有“模板名 + 内容”，还要带归属和使用边界。

建议新增模板表：

- `message_templates`

建议字段：

- `id`
- `template_key`
- `name`
- `description`
- `message_type`
- `owner_scope`
- `owner_tenant_id`
- `audience_type`
- `title_template`
- `summary_template`
- `content_template`
- `action_type`
- `action_target_template`
- `status`
- `meta`

其中：

- `owner_scope`
  - `platform`
  - `tenant`

含义：

- 平台模板：平台统一维护，可用于平台公告、平台待办、团队治理消息
- 团队模板：团队自己的公告或团队内消息模板，只在该团队下可见

### 8.5 模板归属规则

- 平台模板
  - 由平台管理员维护
  - 可发给全体用户、团队管理员、指定团队

- 团队模板
  - 由团队管理员维护
  - 仅能发给所属团队
  - 不允许用于平台级投递

第一阶段即使不先做模板管理页，也建议先把模板归属字段留进模型，避免后续再次改表。

## 9. 权限与可见性

消息中心不建议沿用菜单裁剪语义，而应独立判断：

- 谁能发
- 谁能看
- 谁能处理

建议权限键：

- `system.message.view`
- `system.message.manage`
- `system.message.send`
- `system.message.template.manage`

其中：

- 普通用户默认只有 `view`
- 平台管理员有 `manage/send`
- 团队管理员后续可有团队域发送权限，但第一阶段先不开放后台发信页

## 10. 接口设计

第一阶段建议最少补这些接口：

### 10.1 用户侧接口

- `GET /api/v1/messages/inbox/summary`
- `GET /api/v1/messages/inbox`
- `GET /api/v1/messages/inbox/:deliveryId`
- `POST /api/v1/messages/inbox/:deliveryId/read`
- `POST /api/v1/messages/inbox/read-all`
- `POST /api/v1/messages/inbox/:deliveryId/todo-action`

### 10.2 管理侧接口

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `GET /api/v1/messages/:id`
- `POST /api/v1/messages/:id/publish`
- `POST /api/v1/messages/:id/revoke`

第一阶段即使不做完整后台发信页，也建议先把管理接口边界定好。

## 11. 推送策略

第一阶段：

- 登录后拉一次摘要
- 打开面板时拉消息列表
- 处理动作后局部刷新

第二阶段：

- 接入 websocket 推送未读数变化
- 接入新消息提醒

第三阶段：

- 接入更细的业务事件流
- 支持后台实时协作消息

也就是说：

- 先拉模式
- 再推模式

不要一上来就让消息系统绑定 websocket 才能用。

## 12. 与当前项目的推荐落地顺序

### 12.1 第一阶段

- 保留右上角三栏交互
- 用正式接口替换 mock
- 只做“收件箱查看 + 已读 + 待办处理”
- 新增完整消息中心页 `#/workspace/inbox`

### 12.2 第二阶段

- 增加平台发送能力
- 增加消息模板
- 增加团队公告与邀请消息

### 12.3 第三阶段

- websocket 未读推送
- 更细粒度业务事件接入
- 团队管理员发送能力

## 13. 当前建议结论

一句话定稿：

- 右上角继续保留 `通知 / 消息 / 待办`
- 消息发送者必须支持 `系统 / 用户 / 服务`
- 后端采用 `消息主表 + 用户收件箱表`
- 第一阶段先做拉取式真实收件箱和完整消息中心，不先做复杂推送

这套设计和当前项目最匹配，改动顺序也最稳。

## 15. 2026-03-29 富文本发送与统一收件页调整

基于当前产品方向，消息系统继续收口为：

- 平台侧保留一个统一发信页：`#/system/message`
- 团队侧补一个团队管理员发送页：`#/team/message`
- 用户侧统一使用 `#/workspace/inbox` 作为消息收件页

本轮补充约束：

- 发信页中的摘要、正文统一改为富文本编辑
- 不再单独配置动作按钮，内部跳转和外链统一放入正文超链接
- 收件页列表和右上角消息面板只显示纯文本摘要，避免富文本标签直接泄露到列表
- 收件页详情直接渲染富文本正文
- 当消息存在团队上下文时，消息列表、右上角面板和详情页都显示团队标识，帮助用户识别“来自哪个团队”或“作用于哪个团队”

当前结论：

- 发送侧采用“同组件双页面”：
  - 平台消息页挂到系统管理菜单
  - 团队消息页挂到团队管理菜单
- 不做“一个页面实体挂两个菜单”，而是两条页面注册记录复用同一个发送组件
- 收件侧继续做强，把详情渲染、团队标识和统一入口做好

## 16. 2026-03-29 模板管理与发送记录落地

在消息发送和统一收件页稳定后，消息系统继续补齐了模板管理和发送记录两条后台链路：

- 平台模板页：`#/system/message-template`
- 平台发送记录页：`#/system/message-record`
- 团队模板页：`#/team/message-template`
- 团队发送记录页：`#/team/message-record`

本轮实现原则：

- 平台页和团队页继续采用“多页面记录复用同一组件”的模型
- 不做“一页双挂菜单”，保持页面注册、菜单归属和权限继承关系单一
- 团队页默认带当前团队上下文；若当前账号没有团队归属，则页面入口不会出现在菜单中

模板管理页当前能力：

- 平台侧可维护平台模板
- 团队侧可查看平台模板，也可维护当前团队模板
- 团队页中的平台模板只读展示，不允许在团队侧直接改平台模板
- 模板编辑继续使用富文本，摘要模板和正文模板都支持富文本内容

当前模板归属规则：

- 平台模板：`scope_type = platform`
- 团队模板：`scope_type = team`
- 团队模板键会自动带团队前缀，避免不同团队模板键冲突
- 团队模板的发送对象固定收口到团队用户，不允许在团队侧改成平台范围

发送记录页当前能力：

- 按关键字筛选
- 按消息类型筛选
- 按发送对象筛选
- 查看每次发送的投递汇总：
  - 总投递数
  - 已读数
  - 未读数
  - 待办未处理数
- 查看发送详情，包括标题、富文本摘要、富文本正文、发送对象和来源团队

当前管理侧接口补充为：

- `GET /api/v1/messages/templates`
- `POST /api/v1/messages/templates`
- `PUT /api/v1/messages/templates/:templateId`
- `GET /api/v1/messages/records`

当前页面注册补充为：

- `system.message.template.manage`
- `system.message.record.manage`
- `team.message.template.manage`
- `team.message.record.manage`

当前菜单补充为：

- 平台：
  - `系统管理 / 接口与注册 / 消息模板`
  - `系统管理 / 接口与注册 / 发送记录`
- 团队：
  - `团队管理 / 消息发送`
  - `团队管理 / 消息模板`

## 20. 2026-03-29 接收组第二阶段：角色与功能包规则

在第一阶段的“指定用户 + 接收组”之上，第二阶段继续把接收组扩到两类规则：

- `role`
- `feature_package`

这一轮仍然坚持一个原则：

- 发送页只负责选对象
- 接收组负责承载规则

也就是说，角色和功能包规则不会直接加到发送页表单，而是继续通过接收组组织。

### 20.1 规则语义

平台侧：

- `role`
  - 按平台角色命中用户
  - 基于平台 `user_roles + roles(tenant_id IS NULL)` 匹配
- `feature_package`
  - 按平台有效功能包命中用户
  - 基于 `platform_user_access_snapshots.expanded_package_ids` 匹配

团队侧：

- `role`
  - 按当前团队角色命中成员
  - 同时覆盖：
    - 团队成员身份角色 `tenant_members.role_code`
    - 当前团队附加角色 `user_roles(tenant_id = current_team)`
- `feature_package`
  - 按当前团队有效功能包命中成员
  - 基于 `team_role_access_snapshots.expanded_package_ids` 找出命中功能包的角色，再反推当前团队成员

### 20.2 这样设计的原因

- 平台功能包命中应按“有效功能包”计算，而不是只看直接绑定，否则会漏掉从角色继承来的功能包
- 团队功能包更不能只看团队开通表，因为最终能不能用，仍然和团队角色快照绑定
- 因此这轮直接复用已有访问快照，而不是重新拼一套功能包有效性规则

### 20.3 前端配置变化

接收组页新增两类规则选择：

- 按角色命中
- 按功能包命中

具体表现：

- 平台接收组页可选平台角色、平台功能包
- 团队接收组页可选当前团队角色、团队功能包
- 接收组列表和编辑抽屉会回显角色名、功能包名，不只显示编码

消息发送页新增两类发送对象：

- 接收组中的角色规则
- 接收组中的功能包规则

注意：

- 这两个发送对象仍然要选接收组
- 系统只会展开接收组中对应类型的规则，不会把同组内其他规则一起带出

### 20.4 当前边界

第二阶段仍未开放：

- 标签匹配
- 多条件组合
- 条件表达式
- 动态人群预览

这些都继续保留在接收组模型的扩展方向中，不在本轮展开。

## 21. 2026-03-29 发送记录详情与来源审计

随着接收组开始支持角色、功能包等规则，发送记录不能再只停留在“总共发了多少人”。

这一轮补的目标是：

- 让发送记录页可以展开到单个投递对象
- 明确每个接收人来自哪个接收组、哪条规则
- 为后续“规则命中审计”“投递失败追踪”保留稳定字段

### 21.1 后端落库策略

在 `message_deliveries.meta` 中写入投递来源快照：

- `source_group_id`
- `source_group_name`
- `source_rule_type`
- `source_rule_label`

同时保留原有：

- `recipient_username`

这样做的原因是：

- 历史发送记录需要按发送当时的命中来源展示
- 接收组名称、角色名、功能包名后续可能会变
- 如果只靠实时回查配置表，历史审计结果会漂移

### 21.2 详情接口

新增接口：

- `GET /api/v1/messages/records/:recordId`

返回结构分成两层：

- 发送记录主信息
- `deliveries` 投递明细

投递明细至少包含：

- 接收用户
- 接收团队
- 已读 / 未读 / 待办状态
- 最近动作时间
- 来源接收组
- 来源规则类型
- 来源规则标签

### 21.3 前端展示策略

发送记录页继续保留列表摘要，但详情抽屉改成审计视图：

- 顶部仍看发送标题、类型、对象、发送范围
- 中部增加投递明细区
- 每条投递展示：
  - 用户
  - 团队
  - 当前状态
  - 来源组
  - 来源规则
  - 最近动作时间

这样可以直接回答：

- 这次消息到底发给了谁
- 某个用户为什么会收到
- 他是通过接收组、角色规则还是功能包规则命中的

### 21.4 当前边界

这一轮仍然没有展开：

- 命中链完整树形追踪
- 发送失败重试
- 规则命中前预估人数快照
- 发送记录导出

当前阶段先把“已发出后的来源审计”做稳，再继续补更重的追踪能力。

### 21.5 目标项级来源快照

在仅有“来源接收组 + 规则类型”之后，这一轮继续把来源再下钻到目标项：

- `source_target_id`
- `source_target_type`
- `source_target_value`

它们的用途是：

- `source_target_id`
  - 标识接收组中的具体目标项记录
- `source_target_type`
  - 标识目标项类型，例如 `user / tenant_users / tenant_admins / role / feature_package`
- `source_target_value`
  - 标识目标项关键值，例如用户 ID、角色编码、功能包键

这样发送记录详情不仅能回答“来自哪个接收组”，还能回答：

- 是接收组里的哪条目标项命中的
- 命中的关键条件是什么

前端当前展示策略：

- 团队范围目标项优先展示团队名
- 角色规则展示角色编码
- 功能包规则展示功能包键
- 用户规则展示目标用户 ID

后续如果要继续做易读化，可再把角色编码、功能包键替换成名称快照。
