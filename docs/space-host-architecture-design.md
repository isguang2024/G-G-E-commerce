# 空间与可配置多子域架构设计

> 基线时间：2026-03-30。目标不是立即拆多子域，而是先把“空间模型 + Host 配置模型 + 菜单/页面/权限装载方式”设计稳定，做到单域可运行、多子域可配置启用。

## 1. 设计目标

本项目未来需要承载的平台规模，不适合继续把所有功能、菜单、页面都堆在同一个默认后台壳层里。

但现阶段也不适合直接强制拆成多子域名，因为：
- 当前核心问题还在空间边界、菜单组织、页面注册、权限装载与上下文切换
- 子域名只是承载手段，不是业务模型本身
- 如果先拆域、后补模型，后续返工成本会更高

因此本设计的目标是：
- 默认继续支持单域运行
- 允许通过配置启用多子域
- 平台空间、团队空间、个人空间、未来业务空间共用一套稳定模型
- 菜单、页面、权限、消息、快捷入口都基于“空间”感知，而不是基于硬编码域名判断

一句话：

`先稳定空间模型，再让 Host/Subdomain 变成一个可配置部署策略。`

## 2. 核心结论

### 2.1 不强制多子域

系统必须支持：
- 单域模式
- 主域 + 若干子域模式
- 后续业务线子域扩展

但多子域不应该写死为唯一方案。

### 2.2 先定义空间，再定义域名

优先级顺序必须是：

1. 空间模型
2. 菜单与页面归属
3. 权限准入模型
4. Host 配置模型
5. 是否启用子域名

如果顺序反过来，后面会出现：
- 域名拆好了，但页面不知道自己属于哪个空间
- 菜单仍靠权限裁剪硬隐藏
- 页面直达、面包屑、激活菜单、快捷入口、消息来源都混乱

### 2.3 域名职责要单一

未来推荐的职责边界：
- 域名：决定进入哪个空间外壳
- 菜单：决定空间内显示哪些模块入口
- 页面：决定访问单元、面包屑、父链、挂接关系
- 权限：决定准入、可见性、动作控制
- 消息：决定投递范围、来源空间、接收上下文

不要让域名去承担菜单组织，也不要让菜单去承担空间切换。

## 3. 空间模型

## 3.1 需要显式存在的空间

第一阶段建议稳定四类空间：

- `platform`
  - 平台治理空间
  - 负责租户、权限、菜单、页面、API、平台角色、系统配置

- `team`
  - 团队工作空间
  - 负责团队成员、团队角色、团队功能包、团队消息、团队业务页面

- `personal`
  - 个人空间
  - 负责个人中心、个人消息、个人偏好、个人任务

- `shared`
  - 共享页面空间
  - 允许某些页面在多个空间中复用，但仍要明确宿主空间和权限语义

## 3.2 每个空间必须回答的四个问题

每个空间都必须明确：
- 谁能进入
- 进入后加载哪套菜单
- 页面属于谁
- 接口按哪个上下文运行

例如：

| 空间 | 谁能进 | 菜单来源 | 主要上下文 |
|---|---|---|---|
| platform | 平台管理员/具备平台权限用户 | 平台菜单树 | 无 `X-Tenant-ID` |
| team | 团队成员/团队管理员 | 团队菜单树 | 必须带 `X-Tenant-ID` |
| personal | 已登录个人用户 | 个人页集合 | 用户自身 |
| shared | 取决于宿主空间 | 不单独装菜单 | 跟随宿主 |

## 4. Host / 子域名模型

## 4.1 设计原则

Host 模型必须满足：
- 不配置子域时，系统也能完整运行
- 配置子域后，能够把特定空间绑定到特定 Host
- 同一空间可以先运行在主域，再迁移到子域
- 前端和后端都不应大量写死 `platform.xxx.com` 这类常量

## 4.2 推荐抽象

建议未来采用“空间配置 + Host 映射”的方式，而不是只配置域名字符串。

推荐配置维度：

### `space_definition`
- `space_key`
- `space_type`
- `name`
- `description`
- `status`
- `context_mode`
- `default_menu_scope`
- `default_page_scope`

### `space_host_binding`
- `space_key`
- `host`
- `subdomain`
- `scheme`
- `enabled`
- `is_primary`
- `route_prefix`
- `cookie_scope_mode`
- `auth_mode`
- `login_host`
- `callback_host`
- `cookie_domain`

### `space_entry_policy`
- `space_key`
- `default_landing_route`
- `allow_direct_access`
- `allow_cross_space_jump`
- `login_redirect_priority`

这样做的好处是：
- 子域名只是 `space_host_binding` 的一种配置结果
- 单域模式下也能沿用同一套空间模型
- 后续要切 `admin`、`team`、`me` 子域，不需要重写页面和菜单模型

当前已经落地的 Host 绑定策略字段包括：
- `scheme`
  - `http / https`
- `route_prefix`
  - 用于同一 Host 下的路径前缀兼容，例如 `/admin`
- `auth_mode`
  - `inherit_host`
  - `centralized_login`
  - `shared_cookie`
- `login_host`
  - 统一登录 Host，仅在 `centralized_login` 下使用
- `callback_host`
  - 登录回调 Host，仅在 `centralized_login` 下使用
- `cookie_scope_mode`
  - `inherit / host_only / parent_domain`
- `cookie_domain`
  - Cookie 域，仅在 `shared_cookie` 下使用

这些字段当前只作为可配置底座，不会强制改变现有单域登录方式；默认仍按当前 Host 与 Bearer Token 流程运行。

当前前端仍使用 `hash` 路由，因此跨 Host 目标地址也按这一规则生成：
- 默认根路径：`https://host/#/route-path`
- 带路由前缀：`https://host/prefix/#/route-path`

这意味着：
- 同 Host、同协议、无前缀时，继续使用前端 `router.push`
- Host 改变、协议改变或配置了 `route_prefix` 时，统一走整页跳转
- 这样可以同时兼容当前单域运行和未来多 Host 部署

## 4.3 推荐支持的部署模式

### 模式 A：单域模式

示例：
- `app.example.com`

特点：
- 所有空间都走同一个 Host
- 前端根据空间上下文、菜单树和页面中心做区分
- 最适合当前阶段

### 模式 B：主域 + 平台/团队子域

示例：
- `admin.example.com`
- `team.example.com`
- `app.example.com`

特点：
- 平台治理和团队工作区开始从壳层入口上分开
- 适合中期增长阶段

### 模式 C：多业务空间子域

示例：
- `admin.example.com`
- `team.example.com`
- `crm.example.com`
- `ops.example.com`

特点：
- 更适合成熟阶段
- 前提是空间、权限、菜单、页面治理已稳定

## 5. 页面与菜单的未来优化点

## 5.1 页面中心必须增加空间归属能力

建议未来 `ui_pages` 或页面注册层补充这些概念：
- `space_type`
- `host_strategy`
- `entry_mode`
- `space_visibility`
- `cross_space_access`

建议语义：

- `space_type`
  - `platform / team / personal / shared`

- `host_strategy`
  - `inherit`
  - `main_domain_only`
  - `subdomain_preferred`
  - `subdomain_only`

- `entry_mode`
  - `menu_page`
  - `independent_page`
  - `inner_page`
  - `workflow_page`

## 5.2 菜单不做空间切换器，只做空间内导航

菜单的未来职责应该固定为：
- 导航树
- 分组
- 顺序
- 主页面挂接
- 权限可见性

不建议让菜单承担：
- 当前空间切换
- 数据范围切换
- 团队切换主逻辑

推荐未来的装载顺序：

1. 根据当前空间决定加载哪套菜单集合
2. 再根据权限裁剪菜单
3. 再根据页面挂接关系补完整路由与面包屑

也就是：

`空间过滤 -> 权限裁剪 -> 菜单渲染 -> 页面挂接`

## 6. 权限体系未来要补的关键点

未来一定要把权限拆成两层半：

### 第一层：空间准入
- 能不能进入平台空间
- 能不能进入某个团队空间
- 能不能进入某个业务空间

### 第二层：菜单可见
- 进入空间后，能不能看到某个菜单入口

### 第三层：页面动作
- 进入页面后，能不能做某个动作

当前项目已经有菜单裁剪、页面权限、功能包、权限键、快照，基础不错。

下一步真正要补的是：
- 空间准入模型
- 页面权限继承在多空间下的稳定规则
- 跨空间跳转时的权限判断

## 7. 消息、快捷入口与个人页要同步空间化

这是后续很容易漏掉、但实际很关键的三块。

## 7.1 消息系统

消息系统未来要显式带上：
- 来源空间
- 宿主空间
- 来源团队/平台
- 可跳转目标空间

原因：
- 平台消息、团队消息、个人消息语义不同
- 如果未来启用子域名，消息中心点击跳转可能需要跨 Host

## 7.2 快捷入口

快捷入口未来不应该只是“路由名或链接列表”，而应该具备：
- 空间归属
- 推荐级别
- 是否跨空间可见
- 是否允许跨 Host 打开

当前已经开始收口消息工作台相关重复入口，方向正确。

## 7.3 个人中心

个人中心已经从菜单中抽离到页面中心，这个方向正确。

未来建议继续保持：
- 个人页不挂主菜单
- 个人页属于 `personal` 或 `shared` 空间
- 从用户头像菜单或独立入口进入

## 8. 分阶段开发计划

这里不直接写“改什么代码”，而是按阶段控制风险。

### 第 0 阶段：设计收口

目标：
- 明确空间模型
- 明确 Host 配置模型
- 明确页面中心要补哪些字段
- 明确菜单装载顺序

产出：
- 本文档
- 后续字段设计清单

### 第 1 阶段：只加模型，不改默认行为

目标：
- 页面中心补空间字段
- 菜单/路由装载层补空间过滤入口
- 保持单域模式默认不变

完成标准：
- 系统行为不变
- 但代码结构已经能感知空间

### 第 2 阶段：前端壳层支持空间感知

目标：
- 前端根据当前空间装载不同菜单树
- 页面与面包屑可以根据空间工作
- 登录后落点策略支持按空间选择

完成标准：
- 单域模式下，平台/团队/个人空间已经能明确分层

### 第 3 阶段：配置化 Host 绑定

目标：
- 增加 Host 配置模型
- 前后端都支持根据 Host 识别目标空间
- 但默认仍然可以不启用子域名

完成标准：
- 同一套系统可跑单域，也可跑主域 + 子域

### 第 4 阶段：跨空间跳转

目标：
- 消息、快捷入口、页面跳转支持跨空间导航
- 需要时可以跨 Host 跳转

完成标准：
- 消息和快捷入口不再只考虑当前域

### 第 5 阶段：成熟期治理

目标：
- 审计
- 空间配置后台
- 子域部署规范
- Cookie / 登录态策略统一

## 9. 当前最值得优先做的优化点

按优先级排序，我建议是：

1. 先补“空间模型”专题文档与字段设计
2. 给页面中心补空间归属字段
3. 路由装载增加空间过滤入口
4. 菜单装载改成“空间过滤后再权限裁剪”
5. 登录后落点策略改为空间优先
6. 消息系统补来源空间与目标空间语义
7. 快捷入口补空间归属与跨空间策略
8. 最后再启用 Host / 子域配置

## 10. 当前不建议马上做的事

当前不建议立即做：
- 直接拆多个前端项目
- 直接按功能模块拆多个子域
- 为每个空间单独做一套权限体系
- 为子域跳转提前引入复杂网关逻辑

这些都会让复杂度过早爆炸。

## 11. 最终建议

未来大项目确实应该支持：
- 多空间
- 可配置多子域
- 平台/团队/个人/业务线分层

但当前最稳的路线不是“立刻多子域”，而是：

`先把单域系统做成空间化架构，再把多子域变成一个可选部署能力。`

这会让你后续扩张时更稳，也更适合当前项目阶段。

## 12. 当前落地进度

当前后端已经完成第一阶段底座：
- 已有 `menu_spaces` 与 `menu_space_host_bindings` 基础模型
- `menus` 与 `ui_pages` 已补 `space_key`
- 默认 `default` 菜单空间已补齐
- 菜单树与页面运行时已按 Host / 显式空间预留解析入口

这意味着系统现在仍然可以按现有单域单菜单运行，但后续新增菜单空间或 Host 绑定时，不需要推翻现有模型，只需要继续在这个底座上扩展。
