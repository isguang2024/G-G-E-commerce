# 菜单与页面管理设计方案

## 1. 文档定位

本文档定义一套“保留现有菜单系统、补充全局菜单、独立页面管理”的正式方案。

目标不是重写现有菜单，而是：
- 保留现有菜单管理与旧 `inner` 逻辑。
- 在现有菜单中支持“全局菜单”。
- 新增独立“页面管理”，承接数据内页、详情页、编辑页、流程页、全局页。
- 逐步把“菜单 = 页面”的心智收敛为“菜单管入口，页面管访问单元”。

## 2. 背景与问题

当前菜单系统已经承担了：
- 入口导航
- 排序与显隐
- 面包屑辅助
- 功能权限入口控制
- 一部分 `inner` 内页管理

当前问题：
- 新业务模块页面一多，就会出现“每个页面都进菜单管理勾选”的维护负担。
- 数据详情页、编辑页、流程页并不适合成为菜单入口，但又需要面包屑、高亮和访问策略。
- 现有 `inner` 菜单能解决一部分问题，但仍然和导航入口混在一起，长期会让菜单树越来越臃肿。
- 某些全局页面并不需要功能权限，只需要登录后可访问，但当前缺少明确的“全局菜单 / 全局页”模型。

## 3. 设计原则

- 菜单继续存在，且只管理导航入口。
- 页面必须保留前端路由，不采用“只存数据库不注册路由”的模式。
- 页面管理独立于菜单管理，不把所有页面继续塞进菜单树。
- 旧 `inner` 菜单保留，只做兼容和验证，不强制迁移。
- 全局菜单继续在现有菜单管理中配置，不再单独做第二套菜单系统。
- 页面默认继承上级菜单或模块权限，避免逐页配置权限。

## 4. 正式模型

### 4.1 四层职责

- 菜单：导航入口。
- 页面：实际访问单元。
- 权限：能力边界。
- 功能包：模块开通范围。

正式边界：
- 菜单不再等于页面。
- 页面不直接等于权限。
- 权限不直接等于菜单。

### 4.2 菜单分类

现有菜单继续保留，新增正式访问模式：
- `public`
- `jwt`
- `permission`

含义：
- `public`：未登录可访问。
- `jwt`：登录即可访问，不需要功能权限。
- `permission`：按现有菜单功能权限或模块权限校验。

建议放在现有菜单 `meta.accessMode` 中，避免第一阶段大改旧表结构。

### 4.3 页面分类

页面管理正式支持以下类型：
- `inner`
- `global`

可预留：
- `menu`

第一阶段不要求把现有菜单页迁到页面管理，因此页面管理先只管：
- 数据内页
- 详情页
- 编辑页
- 配置页
- 流程页
- 全局页

## 5. 菜单系统设计

### 5.1 全局菜单定义

全局菜单仍然在现有菜单管理中配置。

它和普通菜单的差异，不是“去别处配”，而是：
- 仍可配置名称、图标、排序、父级、显隐。
- 默认 `accessMode = jwt`。
- 不要求绑定功能权限。
- 不依赖功能包展开结果。

适用场景：
- 工作台
- 个人中心
- 消息中心
- 全局搜索
- 帮助中心

### 5.2 旧菜单兼容规则

现有菜单不迁移、不清洗、不改数据结构语义。

兼容规则：
- 菜单未声明 `meta.accessMode` 时，按当前旧逻辑处理。
- 已存在的权限菜单继续按当前权限链工作。
- 已存在的 `inner` 菜单继续按当前方式渲染和访问。

## 6. 页面管理设计

### 6.1 页面主表建议

建议新增正式主表：`ui_pages`

字段建议：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `page_key` | 页面唯一标识，如 `order.detail` |
| `name` | 页面名称 |
| `route_name` | 路由名称 |
| `route_path` | 路由路径 |
| `component` | 组件路径 |
| `page_type` | `inner` / `global` |
| `source` | `seed` / `sync` / `manual` |
| `module_key` | 所属模块，如 `order` |
| `parent_menu_id` | 绑定上级菜单 |
| `parent_page_key` | 绑定上级页面，可选 |
| `active_menu_path` | 页面进入后应高亮的菜单路径 |
| `breadcrumb_mode` | `inherit_menu` / `inherit_page` / `custom` |
| `access_mode` | `inherit` / `public` / `jwt` / `permission` |
| `permission_key` | 页面独立权限键，可选 |
| `inherit_permission` | 是否继承上级权限 |
| `keep_alive` | 是否缓存 |
| `is_full_page` | 是否全屏页 |
| `status` | `normal` / `suspended` |
| `meta_json` | 扩展元数据 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束建议：
- `page_key` 唯一。
- `route_name` 唯一。
- `route_path + page_type` 唯一。

### 6.2 页面管理心智

页面管理只负责以下内容：
- 页面归属
- 路由元数据
- 面包屑策略
- 高亮策略
- 访问模式
- 页面级权限覆盖

页面管理不负责：
- 菜单排序
- 菜单图标
- 侧边栏结构
- 功能包绑定

## 7. 路由设计

### 7.1 页面必须注册前端路由

页面管理不替代前端路由注册。

页面仍然必须进入前端路由系统，原因：
- 可直接 `router.push`
- 刷新后可恢复
- 标签页、缓存、守卫、面包屑可复用现有体系

### 7.2 路由元数据建议

受页面管理接管的页面建议在路由 `meta` 中声明：

```ts
{
  pageKey: 'order.detail',
  managedPage: true,
  moduleKey: 'order'
}
```

说明：
- `pageKey`：前后端对齐的唯一标识。
- `managedPage`：声明该路由进入页面管理链路。
- `moduleKey`：辅助权限继承和批量归属。

## 8. 面包屑与菜单高亮

### 8.1 默认规则

`inner` 页面默认必须绑定一个 `parentMenuId`。

默认面包屑模式：
- `breadcrumbMode = inherit_menu`

默认行为：
- 面包屑 = 上级菜单链 + 当前页面名
- 左侧菜单高亮 = 上级菜单
- 标签页归属 = 上级菜单下的当前页面

### 8.2 可选规则

- `inherit_page`
  - 从上级页面继续向下派生
  - 适合多级流程页
- `custom`
  - 允许极少数特殊页手工定义 breadcrumb

第一阶段建议只正式开放：
- `inherit_menu`

## 9. 权限规则

### 9.1 菜单访问

菜单访问按 `meta.accessMode` 判定：
- `public`
- `jwt`
- `permission`

### 9.2 页面访问

页面访问按 `accessMode` 判定：
- `inherit`
- `public`
- `jwt`
- `permission`

建议规则：
- `global` 页面默认 `jwt`
- `inner` 页面默认 `inherit`
- 只有特殊页才使用独立 `permission_key`

### 9.3 继承判定顺序

当 `accessMode = inherit` 时：
1. 优先继承 `parentPageKey`
2. 其次继承 `parentMenuId`
3. 再其次继承 `moduleKey`
4. 若都不存在，则降级为 `jwt`

### 9.4 正式目标

避免以下模式：
- 每个详情页都单独配权限
- 每个编辑页都单独绑菜单

推荐模式：
- 模块权限控制入口和大类能力
- 页面默认继承模块或上级菜单权限
- 特殊页面再局部覆盖

## 10. 旧 `inner` 菜单兼容策略

现有 `inner` 菜单保留，不改逻辑，只作为兼容层。

运行时建议判定顺序：
1. 若命中页面管理配置，走新页面管理链路
2. 否则若命中旧 `inner` 菜单，走旧逻辑
3. 否则走普通路由默认逻辑

验证策略：
- 第一阶段挑选一批旧 `inner` 页面做对照验证：
  - 详情页
  - 编辑页
  - 配置页
- 新页面管理稳定后，再决定是否迁移旧 `inner`

## 11. 后端接口设计

建议新增模块：`page-management`

接口建议：

- `GET /api/v1/pages`
  - 页面列表
- `POST /api/v1/pages`
  - 新增页面
- `PUT /api/v1/pages/:id`
  - 更新页面
- `DELETE /api/v1/pages/:id`
  - 删除页面
- `GET /api/v1/pages/:id`
  - 页面详情
- `POST /api/v1/pages/sync`
  - 同步页面注册表
- `GET /api/v1/pages/unregistered`
  - 查询未注册页面
- `GET /api/v1/pages/parent-menu-options`
  - 页面管理专用上级菜单候选

辅助接口建议：
- `POST /api/v1/pages/:id/preview-breadcrumb`
  - 预览面包屑
- `POST /api/v1/pages/batch-bind-menu`
  - 批量绑定上级菜单

## 12. 页面同步策略

后端无法像 API 一样直接从 Gin 路由注册里拿到前端页面信息，因此页面同步必须来自前端清单。

建议正式方案：
- 前端维护页面注册清单 `page-manifest`
- 每条受管理页面路由都必须带 `pageKey`
- 页面管理页支持“同步页面”
- 同步来源统一写入 `source = sync`

页面清单字段建议：
- `pageKey`
- `name`
- `routeName`
- `routePath`
- `component`
- `moduleKey`
- `pageType`
- `defaultAccessMode`

第一阶段可接受：
- 手工录入 + 同步并存

## 13. 前端管理页设计

建议新增“页面管理”页面。

列表字段建议：
- 页面名称
- 页面标识
- 页面类型
- 路由路径
- 所属模块
- 上级菜单
- 面包屑模式
- 访问模式
- 权限键
- 来源
- 状态
- 更新时间

筛选项建议：
- 页面类型
- 所属模块
- 上级菜单
- 访问模式
- 来源
- 关键词

操作建议：
- 编辑页面
- 绑定上级菜单
- 配置访问模式
- 配置权限
- 预览面包屑
- 同步页面
- 停用/启用

## 14. 实施阶段建议

### 第一阶段：最小可落地版本

- 菜单支持 `meta.accessMode`
- 现有菜单管理中允许新增全局菜单
- 新增页面管理主表
- 页面类型先开放：
  - `inner`
  - `global`
- 页面管理先支持：
  - 绑定上级菜单
  - `breadcrumbMode = inherit_menu`
  - `accessMode = inherit | jwt | permission`
- 旧 `inner` 菜单原样保留

### 第二阶段：增强版

- 页面清单同步
- 页面级面包屑预览
- 批量绑定上级菜单
- 模块级页面筛选与批量归属
- `inherit_page` 支持

### 第三阶段：统一版

- 评估是否将新建的“数据内页”全部转到页面管理
- 旧 `inner` 菜单逐步只读化
- 最终把菜单系统收口为“入口导航管理”

## 15. 当前明确不做的事

- 不强制迁移现有菜单数据
- 不一次性下线旧 `inner`
- 不把所有页面都重新导入菜单树
- 不让页面管理接管菜单排序和图标
- 不把“页面同步”设计成后端自己扫描前端路由

## 16. 结论

正式建议如下：
- 菜单继续保留，且全局菜单仍在现有菜单管理中配置。
- 全局菜单默认 `accessMode = jwt`，不要求绑定权限。
- 页面管理独立出来，专门管理数据内页和全局页。
- 页面仍然注册前端路由。
- `inner` 页面通过绑定上级菜单来继承面包屑、高亮和默认权限。
- 旧 `inner` 菜单保留，只做兼容与验证。

后续新增功能模块时，推荐流程变为：
1. 注册模块路由
2. 在页面管理中同步或录入页面
3. 只配置少量模块入口菜单
4. 数据内页统一挂靠到模块入口菜单
5. 默认继承模块或上级菜单权限

这套方案可以在不推翻现有菜单系统的前提下，显著降低未来模块扩展成本。
