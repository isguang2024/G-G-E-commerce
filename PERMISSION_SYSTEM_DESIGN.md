# 多租户权限系统完整设计

## 1. 设计目标

本设计面向未来的多租户后台系统，目标是建立一套可长期演进的权限模型，满足以下要求：

- 一个用户可以同时属于多个团队或租户
- 同一用户在不同租户下可以拥有不同角色
- 平台级权限和租户级权限相互隔离，但允许叠加
- 菜单权限、按钮权限、接口权限、数据权限有统一的规则
- 权限冲突有明确裁决逻辑，不依赖模糊约定
- 后续扩展审批、财务、运营、商品、订单等业务时无需推翻现有模型

## 2. 核心原则

### 2.1 认证与权限分离

- 登录只证明“你是谁”
- 当前租户只表示“你正在什么上下文里操作”
- 权限判断则回答“你在这个上下文里能做什么”

这三者不能混在一起。

### 2.2 成员关系与权限角色分离

- `tenant_members` 负责成员归属关系
- `user_roles` 负责角色分配

不要再把复杂权限判断继续堆在 `tenant_members` 上。

### 2.3 菜单、按钮、数据权限分层处理

三类权限虽然都叫“权限”，但语义完全不同：

- 菜单权限：决定用户是否能看见入口
- 按钮权限：决定用户能否执行某个动作
- 数据权限：决定用户能看到和操作哪些数据

三者必须采用不同的计算方式。

### 2.4 优先级只用于“冲突裁决”

`priority` 不应该只是一个展示字段。

如果要保留它，就必须只在存在冲突的场景下使用，例如：

- 多角色对同一个按钮有不同授权结果
- 多角色对同一类数据范围给出不同规则

如果没有冲突模型，`priority` 就不应该参与计算。

## 3. 数据模型

### 3.1 用户表 `users`

职责：

- 表示登录主体
- 保存基础身份信息

关键字段：

- `id`
- `username`
- `email`
- `status`
- `is_super_admin`

### 3.2 租户表 `tenants`

职责：

- 表示一个团队、组织或业务空间

关键字段：

- `id`
- `name`
- `owner_id`
- `status`

### 3.3 成员关系表 `tenant_members`

职责：

- 表达某用户属于哪个租户
- 记录成员状态和加入信息

建议字段：

- `id`
- `tenant_id`
- `user_id`
- `status`
- `joined_at`
- `invited_by`
- `role_code`

建议说明：

- `role_code` 仅用于快速表达成员身份，例如 `team_admin`、`team_member`
- `role_id` 不再作为主权限来源，建议逐步废弃

### 3.4 角色表 `roles`

职责：

- 定义角色模板

建议字段：

- `id`
- `code`
- `name`
- `scope_id`
- `status`
- `priority`
- `sort_order`
- `is_system`

说明：

- `scope_id` 用于区分平台角色、租户角色、未来业务域角色
- `priority` 只参与冲突裁决，不参与普通列表排序

### 3.5 用户角色表 `user_roles`

职责：

- 表达“用户在某个作用域下拥有哪些角色”

建议字段：

- `user_id`
- `role_id`
- `tenant_id nullable`

语义：

- `tenant_id IS NULL`：平台级角色
- `tenant_id = 某租户`：该租户下的角色

建议唯一约束：

- `(user_id, role_id)` 且 `tenant_id IS NULL`
- `(user_id, role_id, tenant_id)` 且 `tenant_id IS NOT NULL`

### 3.6 菜单表 `menus`

职责：

- 定义前端导航资源

关键字段：

- `id`
- `parent_id`
- `path`
- `name`
- `component`
- `sort_order`
- `meta`

推荐 `meta` 字段：

- `requiresTenantContext`
- `alwaysShow`
- `keepAlive`
- `hidden`
- `icon`

### 3.7 角色菜单表 `role_menus`

职责：

- 定义角色拥有哪些菜单入口

字段：

- `role_id`
- `menu_id`

说明：

- 不需要 `tenant_id`
- 因为租户范围已经由 `user_roles.tenant_id` 决定

### 3.8 按钮权限资源表 `permission_actions`

职责：

- 定义可控制的动作权限

建议字段：

- `id`
- `resource_code`
- `action_code`
- `name`
- `description`
- `tenant_scoped`

示例：

- `resource_code = order`, `action_code = create`
- `resource_code = order`, `action_code = refund`
- `resource_code = team_member`, `action_code = invite`

建议唯一约束：

- `(resource_code, action_code)`

### 3.9 角色按钮权限表 `role_action_permissions`

职责：

- 定义角色对某个动作的授权结果

建议字段：

- `role_id`
- `action_id`
- `effect`

其中 `effect` 取值固定为：

- `allow`
- `deny`
- `inherit`

建议说明：

- 一般不建议存 `inherit`
- 默认没有配置时就视为继承
- 如果存储层要显式表达，也要保持语义稳定

### 3.10 数据权限表 `role_data_permissions`

职责：

- 定义角色的数据访问范围

建议字段：

- `role_id`
- `resource_code`
- `scope_type`
- `scope_value`

推荐 `scope_type`：

- `self`
- `department`
- `team`
- `tenant`
- `all`
- `custom`

其中：

- `scope_value` 只在 `custom` 等特殊模式下使用

## 4. 权限域划分

建议把权限分为三层。

### 4.1 平台层权限

面向平台后台和系统管理：

- 用户管理
- 角色管理
- 菜单管理
- 作用域管理
- 团队管理
- 系统设置

来源：

- `user_roles.tenant_id IS NULL`

### 4.2 租户层权限

面向某个团队内部：

- 团队成员管理
- 团队角色分配
- 团队商品管理
- 团队订单管理
- 团队素材管理

来源：

- `user_roles.tenant_id = currentTenantId`

### 4.3 成员关系层

面向成员归属和状态：

- 是否属于某租户
- 是否被冻结
- 是否已加入
- 是否待邀请

来源：

- `tenant_members`

这一层不是完整权限体系，只是权限判断的前置条件。

## 5. 当前租户上下文模型

### 5.1 登录后行为

推荐方案：

- 登录只返回用户身份和 token
- 登录后单独加载可访问租户列表
- 前端保存 `currentTenantId`
- 切换团队时只切换上下文，不重新登录

### 5.2 权限计算输入

每次权限计算至少需要这两个输入：

- `user_id`
- `tenant_id`，可为空

解释：

- 没有 `tenant_id` 时，按平台上下文计算
- 有 `tenant_id` 时，按“平台角色 + 当前租户角色”一起计算

## 6. 菜单权限规则

### 6.1 计算规则

菜单权限采用并集模型。

规则：

1. 取用户在当前上下文中的有效角色集合
2. 聚合这些角色对应的所有菜单
3. 对菜单去重
4. 自动补全上级菜单节点
5. 输出最终菜单树

### 6.2 为什么菜单权限不建议用 `priority`

原因：

- 菜单本质是导航入口
- 只要有任一角色允许用户进入这个页面，就应显示
- 使用 `priority` 去做菜单覆盖，会导致菜单可见性不稳定

所以菜单权限只用：

- 并集
- 去重
- 补父节点

不做“高优覆盖低优”。

### 6.3 团队上下文菜单

对于依赖当前团队的菜单，建议继续使用：

- `meta.requiresTenantContext = true`

行为规则：

- 普通用户无团队时，不显示这类菜单
- 系统管理员不受限制
- 路由守卫保留兜底

## 7. 按钮权限规则

按钮权限必须引入显式冲突裁决模型。

### 7.1 权限效果

每个角色对某个动作的结果只能是：

- `allow`
- `deny`
- `inherit`

### 7.2 推荐裁决规则

推荐固定为：

1. 显式 `deny` 优先于显式 `allow`
2. 显式 `allow` 优先于 `inherit`
3. 如果存在多个显式规则，再比较角色 `priority`
4. 如果 `priority` 一样，则按更严格规则优先

可写成：

- `deny > allow > inherit`

同效果下再比较：

- `priority DESC`

### 7.3 示例

用户有两个角色：

- `finance_admin`，优先级 100，对 `order.refund` 为 `allow`
- `auditor`，优先级 50，对 `order.refund` 为 `deny`

若采用“拒绝优先”规则，则结果为：

- `deny`

如果业务想改成“高优先级覆盖”，则必须明确定义：

1. 先比较 `priority`
2. 再看 `effect`

但我不建议这样设计，因为容易带来安全风险。

更稳的企业系统规则仍然是：

- `deny` 永远最高优先

### 7.4 推荐结论

按钮权限建议固定采用：

- `deny > allow > inherit`

`priority` 只在多个同类型规则冲突时参与比较，不覆盖“拒绝优先”原则。

## 8. 数据权限规则

数据权限不能简单并集，否则很容易出现越权。

### 8.1 建议的数据范围模型

对每个业务资源定义统一的数据范围：

- `self`
- `team`
- `tenant`
- `all`
- `custom`

含义示例：

- `self`：只能看自己创建或归属自己的数据
- `team`：只能看本业务小组数据
- `tenant`：只能看当前租户所有数据
- `all`：可以看平台所有数据
- `custom`：可配置特定组织、成员、标签、类目等

### 8.2 推荐裁决规则

数据权限推荐采用“先看最高优先显式规则，再决定范围”的模式。

建议规则：

1. 收集用户所有有效角色在目标资源上的数据权限规则
2. 过滤掉停用角色
3. 按 `priority DESC` 排序
4. 取最高优先级规则作为主规则
5. 如果同优先级有多个规则，默认取更严格范围

推荐严格度顺序：

- `self` 最严
- `team`
- `tenant`
- `all` 最宽

即同优先级冲突时取最小范围。

### 8.3 为什么不建议直接取并集

例如：

- 角色 A：只能看本人订单
- 角色 B：能看全租户订单

如果做并集，结果会直接放大权限。

这只有在业务明确允许“角色叠加放大数据范围”时才可接受，一般后台系统不建议这样做。

### 8.4 推荐结论

数据权限默认采用：

- 最高优先级优先
- 同优先级取最严格范围

这是最稳的企业后台方案。

## 9. `priority` 的正式定义

建议把 `priority` 明确定义为：

“角色在冲突裁决中的优先级，值越大优先级越高。”

并明确规定：

- 不参与菜单并集计算
- 可参与按钮权限冲突裁决
- 可参与数据权限冲突裁决
- 不等同于列表展示排序

### 9.1 推荐取值建议

- 系统超管：1000
- 平台管理员：900
- 平台运营：700
- 租户管理员：500
- 业务经理：300
- 普通成员：100
- 访客：10

### 9.2 更新规则建议

接口层必须允许：

- 更新为 `0`
- 更新为任意合法整数

不要再使用“只有大于 0 才更新”这种逻辑。

## 10. 权限计算总流程

建议统一为以下流程。

### 10.1 菜单权限

1. 校验用户状态是否可用
2. 读取当前租户上下文
3. 获取有效角色集合
4. 查询菜单权限集合
5. 生成去重后的菜单树
6. 前端再按 `requiresTenantContext` 做展示过滤

### 10.2 按钮权限

1. 校验用户状态
2. 获取有效角色集合
3. 读取目标动作在这些角色上的授权效果
4. 先按 `effect` 裁决
5. 再按 `priority` 裁决
6. 输出单个动作布尔值

### 10.3 数据权限

1. 校验用户状态
2. 获取有效角色集合
3. 读取这些角色在目标资源上的数据范围
4. 按 `priority` 取主规则
5. 同优先级按严格度裁决
6. 转换为最终 SQL 条件或查询约束

## 11. 接口设计建议

### 11.1 菜单权限接口

建议保留：

- `GET /api/v1/menus/tree`
- `GET /api/v1/users/:id/permissions`

说明：

- 前者给当前登录用户使用
- 后者主要给管理后台查看某用户权限结构

### 11.2 按钮权限接口

建议新增：

- `GET /api/v1/auth/actions`

返回当前上下文下可执行的动作编码，例如：

- `order:create`
- `order:refund`
- `team_member:invite`

前端按钮直接按动作编码判断显示与禁用。

### 11.3 数据权限接口

不建议前端直接计算。

建议由后端在每个业务查询接口内自动应用数据范围条件，例如：

- 订单列表
- 商品列表
- 成员列表
- 财务报表

即数据权限应主要由服务端强制控制。

## 12. 前端落地建议

### 12.1 菜单控制

- 菜单由后端树接口返回
- 前端只做轻量过滤
- `requiresTenantContext` 继续在路由层和菜单层双重兜底

### 12.2 按钮控制

建议新增统一指令或 Hook：

- `v-permission="'order:refund'"`
- `useActionPermission('order:refund')`

### 12.3 租户切换

切换租户后需要刷新：

- 菜单
- 动作权限
- 当前页面数据

### 12.4 无团队状态

普通用户无团队时：

- 仍可访问不依赖团队的页面
- 团队上下文菜单隐藏
- 访问团队页面时显示空状态提示

## 13. 分阶段实施方案

### 阶段 1

目标：

- 稳定当前菜单权限体系

内容：

- 保持菜单权限并集模型
- 保留 `user_roles.tenant_id`
- 继续使用 `requiresTenantContext`
- 修正 `priority` 更新逻辑

### 阶段 2

目标：

- 引入按钮权限体系

内容：

- 新增 `permission_actions`
- 新增 `role_action_permissions`
- 前端支持按钮权限指令
- 后端提供动作权限聚合接口

### 阶段 3

目标：

- 引入统一数据权限体系

内容：

- 新增 `role_data_permissions`
- 在核心业务查询里自动应用数据范围
- 建立统一的 Query Filter 注入机制

### 阶段 4

目标：

- 清理历史冗余字段和旧逻辑

内容：

- 逐步废弃 `tenant_members.role_id`
- 弱化 `tenant_members.role_code` 的权限职责
- 把权限主来源统一到 `user_roles + role_*_permissions`

## 14. 推荐结论

如果只给一句最终建议：

- 菜单权限用并集
- 按钮权限用 `deny > allow > inherit`
- 数据权限用“最高优先级优先，同优先级取最严格范围”
- `priority` 只参与冲突裁决，不参与菜单并集
- 平台权限与租户权限统一由 `user_roles(tenant_id)` 承载

这套模型最稳，也最适合你当前这个项目继续往完整多租户权限系统演进。
