# 前端管理页统一规范

> 现状基线：2026-03-27。默认适用于 `frontend/src/views/system/**` 与 `frontend/src/views/team/**`。

## 1. 规则级别

- `Must`：默认必须遵守，除非页面存在明确历史约束且已说明原因。
- `Should`：优先遵守；若有业务特殊性，需在同模块内保持一致。

## 2. 当前前端架构共识

- `Must`：菜单是导航入口，页面是访问单元；详情页、配置页、流程页优先进入页面管理，不再回到菜单树里伪装成“内页菜单”。
- `Must`：菜单管理分组和页面普通分组只影响管理视图，不影响运行时权限、菜单树和访问路径。
- `Must`：页面访问、菜单可见性、动作权限统一消费后端正式接口，不在页面里硬编码角色名做绕过判断。

## 3. 搜索区规范

- `Must`：统一使用 `ArtSearchBar` 承载搜索区。
- `Should`：默认支持折叠；字段顺序为主筛选 -> 关键词/路径 -> 辅助筛选。
- `Must`：按钮顺序固定为 `查询` 在前，`重置` 在后。
- `Must`：搜索区与表格卡片间距统一为 `12px`。

## 4. 页面头部与操作区

- `Must`：头部优先展示页面标题、摘要统计和主操作，不堆整排低频按钮。
- `Must`：列表行操作统一使用 `ArtButtonMore` 三点菜单。
- `Must`：不在表格操作列长期并排放置超过 2 个文本按钮。
- `Must`：危险操作放在菜单末尾，并使用危险色。
- `Should`：操作顺序统一为查看/配置 -> 编辑 -> 状态切换 -> 删除。

## 5. Drawer / Dialog 规范

- `Must`：大表单、绑定关系、权限配置、树选择、批量配置统一使用右侧 `Drawer`。
- `Must`：主配置抽屉统一追加 `config-drawer` 类，复用统一 header/body/footer 间距基线。
- `Should`：轻量确认、预览、小型单字段输入、只读帮助继续使用 `Dialog`。
- `Must`：抽屉顶部工具栏统一为左侧说明/筛选，右侧主操作按钮。
- `Must`：主按钮文案直接使用动作词，如 `新增`、`保存`、`确认`。

## 6. 状态与辅助信息

- `Must`：状态统一使用 `ElTag`。
- `Must`：补充说明使用浅色小字：
  - `color: var(--el-text-color-secondary)`
  - `font-size: 12px`
- `Must`：空值统一显示 `-`。
- `Must`：路径、页面标识、权限键等技术字段优先用等宽或接近等宽风格展示，避免与标题文本混在一起。

## 7. 菜单与页面表单规范

- `Must`：菜单编辑页只配置导航入口、显示、访问模式、管理分组和展示元数据，不再承载“页面内页类型”语义。
- `Must`：页面编辑页围绕以下字段组织：
  - 基础信息
  - 路由与渲染
  - 挂载与归属
  - 访问与行为
- `Should`：页面表单保留“最终路径预览”，帮助用户理解相对路径拼接结果。
- `Must`：当页面 `accessMode = permission` 时，必须显式填写 `permissionKey`；否则前端只做字段收集，不自行推导权限键。

## 8. 路由与权限消费规则

- `Must`：运行时页面通过后端 `/pages/runtime` 与 `/pages/runtime/public` 提供的数据动态注册，不在业务页面重复写第二套访问判定。
- `Must`：页面访问模式统一理解为：
  - `public`：公开访问
  - `jwt`：登录即可
  - `permission`：按权限键校验
  - `inherit`：继承上级菜单或上级页面
- `Must`：需要菜单门槛时，优先消费菜单 `meta` 中的能力配置，不在页面组件内部额外复制逻辑。

## 9. 落地要求

- 新管理页默认按本规范实现。
- 旧页面改造优先处理：
  - 搜索区
  - 头部主操作
  - 操作列
  - 配置抽屉
- 与历史页面冲突时，优先保证同类页面交互一致，再做视觉微调。

## 10. 反例

- 禁止继续把详情页、编辑页塞进菜单树当作导航节点维护。
- 禁止同类页面一部分用抽屉、一部分用大号对话框且没有明确原因。
- 禁止前端按角色名硬编码“显示就等于有权限”。
- 禁止把管理分组、普通分组误当成运行时权限或路由层级。
