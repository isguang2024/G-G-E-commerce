# DataGrid 后台专题

基于 Fluent UI React v9 Storybook 的 `DataGrid` 文档整理。

## 什么时候优先用 DataGrid

- 需要多列结构化数据展示
- 需要排序、选择、可聚焦单元格或更强的交互能力
- 需要比基础 `Table` 更统一的特性集合

官方文档明确指出：

- `DataGrid` 是 `Table` 原语和 `useTableFeatures` 之上的高阶组件
- 它是带意见的组件
- 如果需求与官方示例差异很大，应考虑改用 `Table` + `useTableFeatures` 或自管状态

## 什么时候不要先用 DataGrid

- 只有单列内容
- 布局更像信息列表或卡片列表
- 单元格结构非常自由，已经偏离表格语义
- 你已经知道自己需要绕过大部分默认行为

## 后台默认结构

- 标题区
- 筛选区
- 工具栏
- `DataGrid`
- 详情 `Drawer` 或分页区

## 可访问性与结构要求

- 始终带 `DataGridHeader`
- 如果前面有可见标题，用 `aria-labelledby` 指向标题
- 如果没有可见标题，用 `aria-label`
- 给表格容器设置 `min-width`，避免高缩放和窄屏下塌陷
- 不要覆写 DataGrid 相关角色语义

## 常见后台模式

### 资源列表

- 文件、工单、订单、角色、菜单、用户、审计记录这类多列对象列表优先用 `DataGrid`
- 状态列用 `Badge`、`Tag`
- 行级低频操作进菜单

### 选择式列表

- 批量操作、行勾选、单选绑定、资源挑选对话框适合 `DataGrid`
- 如果是单选，保持交互非常明确，不要把整行多种点击行为混在一起

### 大数据量

- 需要虚拟化时优先沿官方示例做
- 如果虚拟化后需求过度定制，评估回退到更低层实现

## 不要这样用

- 不要用 `DataGrid` 显示本质上是描述性单列内容的页面
- 不要把复杂表单直接塞进大量单元格
- 不要在同一个单元格里叠加多个语义冲突的可点击元素

## 与 Table 的决策规则

- 需求标准化、交互常见、希望快速交付：优先 `DataGrid`
- 布局自由、行为特殊、组合复杂：优先 `Table`

## 实施建议

- 先从官方 `Default`、`Sort`、`Select`、`Virtualization` 示例心智出发
- 后台页面最常见的是“排序 + 选择 + 行操作 + 详情抽屉”组合
- 若需求已经明显超出官方示例，尽早切回 `Table` 路线，不要在 `DataGrid` 上硬拗
