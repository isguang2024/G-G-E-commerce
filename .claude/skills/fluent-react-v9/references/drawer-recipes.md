# Drawer 后台专题

基于 Fluent UI React v9 Storybook 的 `Drawer` 文档整理。

## 三种抽屉心智

- `OverlayDrawer`
  覆盖在页面之上，默认更强打断，底层基于 `Dialog`
- `InlineDrawer`
  在容器内联出现，适合与主内容并排
- `Drawer`
  可在 overlay 与 inline 之间切换，常用于响应式

## 选型规则

### OverlayDrawer

适用于：

- 需要用户集中注意力
- 表单、配置、审批确认、复杂编辑
- 窄屏下需要覆盖主内容

避免：

- 只是查看辅助信息
- 页面本来已经有稳定的右侧属性区

### InlineDrawer

适用于：

- 详情、属性、说明、导航辅助
- 不需要完全打断主任务
- 主列表和详情要同时可见

避免：

- 需要强确认的不可逆操作

### Drawer

适用于：

- 桌面端内联、移动端覆盖
- 同一业务需要响应式切换两种模式

不要：

- 没有切换需求却为了“保险”默认用组合版 `Drawer`

## 后台常见模式

### 列表页右侧详情

- 桌面优先 `InlineDrawer`
- 移动端可切 `OverlayDrawer`
- 让主列表保持上下文，不强制跳走

### 配置编辑

- 中大型配置表单优先 `OverlayDrawer`
- 头部说明、主体表单、底部操作分区清楚

### 审核与处理

- 对象详情与处理操作放在 Drawer 内
- 把风险说明和提交动作靠近底部操作区

## 最佳实践

- 抽屉只承担一个主任务
- 标题、说明、操作区要稳定
- 大表单不要挤成弹窗感
- 让返回主工作区的路径直观

## 不要这样用

- 不要把简单二次确认升级成大抽屉
- 不要在抽屉里再套复杂多层浮层当主流程
- 不要让抽屉既像导航又像主编辑器又像帮助中心

## 响应式建议

- 宽屏优先 inline，保留上下文
- 窄屏优先 overlay，避免挤压主内容
- 若同一页面确实需要两种模式，才用组合版 `Drawer`
