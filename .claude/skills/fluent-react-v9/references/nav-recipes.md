# Nav 后台专题

基于 Fluent UI React v9 Storybook 的 `Nav` 文档整理。

## 什么时候优先用 Nav

- 后台模块导航
- 工作台一级或二级信息架构
- 需要稳定、可折叠、可选择的应用级导航

官方文档强调：

- `Nav` 提供最多两级嵌套
- 更适合应用导航，而不是任意深度对象树

## 什么时候不要用 Nav

- 层级深于两级
- 数据本质是资源树、对象树、文件树、菜单树
- 需要大量节点操作、拖拽、层级编辑

这类情况优先考虑 `Tree`。

## 后台默认用法

- 左侧模块导航优先 `Nav`
- 一级是业务域，二级是模块入口
- 选中态明确，只保留一个当前主路径

## 可访问性要求

- 会改变 URL 的项要带 `href`
- 折叠按钮要有可访问名称
- 内联折叠态切换按钮要补 `aria-expanded`
- 不要把“展开/收起”和“跳转链接”混在同一语义按钮上

## 常见模式

### 应用侧栏

- 适合平台后台、运营后台、设置中心
- 顶部保留品牌与全局动作
- 侧栏只负责导航，不堆业务按钮

### 分组导航

- 一级模块 + 二级功能页
- 若某组只是一层标签，不要强行做可展开项

### 响应式

- 桌面可用 inline 侧栏
- 窄屏切 hamburger 或 overlay 导航

## 不要这样用

- 不要把对象操作树做成 `Nav`
- 不要用三级、四级导航硬塞进 `Nav`
- 不要把同一项同时做成链接和展开器

## 与 Tree 的决策规则

- 应用导航、模块导航：`Nav`
- 对象层级、结构树、菜单树：`Tree`
