# Tree 后台专题

基于 Fluent UI React v9 Storybook 的 `Tree` 文档整理。

## 什么时候优先用 Tree

- 数据天然是父子层级
- 需要展开、折叠、选择、定位层级节点
- 场景是菜单树、资源树、文件树、分类树、组织结构树

官方文档强调：

- `Tree` 用于可折叠的层级列表
- 若交互更复杂或数据量很大，可考虑 `FlatTree`

## 什么时候不要用 Tree

- 数据其实只是导航入口
- 数据不具备明显层级关系
- 页面更适合表格、列表或卡片

## 后台常见模式

### 菜单管理

- 左侧 `Tree`
- 右侧详情面板或 `Drawer`
- 顶部工具栏放创建、刷新、筛选和视图动作

### 文件或资源结构

- 树承载定位
- 详情区承载属性与操作
- 不要把复杂编辑直接塞回树节点里

### 组织结构或分类维护

- 树用于层级浏览
- 表单用于编辑节点
- 需要复杂 manipulation 时评估 `FlatTree`

## 最佳实践

- 树根要有可访问名称，例如 `aria-label`
- 层级非常深时，按官方建议考虑自定义缩进样式
- 节点展示信息要克制，避免一行里堆太多动作
- 高价值动作靠近详情区或工具栏，而不是全塞节点尾部

## 不要这样用

- 不要把应用导航与对象树职责混在一起
- 不要把多种互相冲突的点击行为放在同一节点主区域
- 不要在树节点内部塞复杂表单

## Tree 与 FlatTree

- 常规层级展示和操作：先用 `Tree`
- 大量节点、复杂操作、性能要求更高：再评估 `FlatTree`
