# Fluent 2 模式与系统规则

用于把设计原则翻译成可执行的前端界面策略。

## Handoffs

适用场景：

- 用户从一个页面切到另一个更强功能的页面
- 从聊天、摘要、概览跳到编辑器、表单、文档或详情页
- 需要保留上下文继续工作

规则：

- 交接必须引导顺滑，不让用户自己猜下一步。
- 交接必须保留必要上下文，不把不相关内容一起带过去。
- CTA 文案尽量用官方常见模式：`Create in ...`、`Open in ...`、`Continue in ...`、`Try in ...`
- 交接说明要短、完整、可回溯。

## Wait UX

适用场景：

- 加载、上传、生成、同步、批量提交、报表计算

规则：

- 清楚说明当前在做什么，而不是只放一个转圈。
- 能给确定进度就不要只给不确定进度。
- 长等待要说明对象、阶段或下一步，降低不确定性。
- 不要用拟人化文案误导用户系统能力。

## Motion

- 动效必须有功能目的。
- 动效要帮助用户理解层级、过渡和状态变化。
- 统一时长、节奏和方向，避免页面里出现多套动效语言。
- 后台产品里，动效的任务是增强理解，不是制造表演感。

## Design tokens

- 颜色、间距、圆角、阴影和字号优先用 token，不直接写死局部值。
- 品牌定制优先通过 token 收口，而不是单页覆写。
- 遇到跨页面不一致时，先查 token 层，再查局部实现。

## Accessibility

- Fluent 2 不是“好看优先”，而是把可读、可操作、可理解一起做到位。
- 文本缩放到 200% 时仍应尽量避免裁切。
- 信息层级要支持快速扫读。
- 文案和状态不要只靠颜色传递。
