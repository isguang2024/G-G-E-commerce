# Change Wrapup（仓库内定制版）

用于任务结束时收口，不负责过程记录。本版与 V5 重构期对齐，落点为 `docs/V5_REFACTOR_TASKS.md`，不再维护独立的 change-log。

## 执行原则

- 只写对用户有价值的结论、影响、验证和下一步
- 先写结果，再写补充
- 控制篇幅，避免把最终回复写成完整变更日志
- 不重复罗列低价值实现细节

## 边界

- 本技能只负责最终收尾
- 过程中的报错、纠正、试错记录不进入最终回复
- 不创建新的日志文件；所有阶段性沉淀都进 `docs/V5_REFACTOR_TASKS.md`
- 不再创建或写入 `docs/change-log.md`

## 大型改动判定

满足任一条件即视为大型改动：

- 跨模块改动（涉及 ≥ 2 个 `backend/internal/modules/*` 或前后端同时改）
- 涉及数据库 schema 变更、迁移或 seed
- 涉及 OpenAPI spec（`backend/api/openapi/`）变更
- 涉及权限模型 / evaluator / casbin 策略
- 涉及核心流程、接口契约、状态管理
- 修改超过 10 个文件
- 包含重构、迁移或架构调整
- 需要额外说明兼容性、风险、测试计划或后续收尾事项

不满足以上任一条件即为小改动，按"小改动输出"走，不写两段式总结、不写阶段进度。

## 输出方式

小改动：

- 用 1 至 2 句交代结果
- 有验证就写验证；未验证就明确写未验证

大型改动：

```md
## 本次改动
- 已完成的关键修改
- 这样改的原因
- 影响范围、验证结果或已知限制

## 下次方向
- 下一步最有价值的推进项
- 待确认问题、风险点或可继续优化项
```

约束：

- 每段 2 到 4 条
- "本次改动"只写已完成内容
- "下次方向"只写下一轮值得继续推进的事项

## 阶段进度同步

仅在大型改动时执行：追加写入 `docs/V5_REFACTOR_TASKS.md` 末尾的 `## 阶段进度` 小节。

- 若 `## 阶段进度` 标题不存在，在文件末尾创建一次
- 在该标题下追加新的 `### YYYY-MM-DD 任务简述（Phase X）` 块，不修改其他章节
- 同一天多次收尾按追加方式叠加，不覆盖已有块
- UTF-8 编码

格式：

```md
### YYYY-MM-DD 任务简述（Phase X）

**本次改动**
- ...

**下次方向**
- ...
```
